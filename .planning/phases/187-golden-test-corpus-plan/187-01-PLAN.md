**Section 2 — Test Corpus (TEST-02):**

Define 30+ test cases. The first 10 cases are the worked examples from `02-compilation-pipeline.md` §10 (Examples 1-10). These are the backbone per D-04. Present each case as a structured data object with these fields:
- `name`: descriptive string (e.g., `"single-calc-derived-column"`)
- `category`: which explorer/category is being tested (`formulas/calculations`, `formulas/filters`, `formulas/sorts`, `marks/conditional`, `audits/anomaly`, `audits/validation`, `cross-category`)
- `chipArrangement`: the chip well state (conforming to ChipWellOutputContract shape from 06-chip-well-geometry.md §9) — list of chips per well with DSL fragments and type signatures
- `viewContext`: optional GROUP BY from view explorer (null if no grouping)
- `expectedSql`: the exact SQL string (whitespace-normalized per D-10)
- `expectedBindValues`: the bind value array
- `expectedResult`: for annotation cases (Marks/Audits), the expected annotation Map entries; for SQL cases, the expected row objects array (sort then deep-equal per D-11)
- `orderSensitive`: boolean — true when the test includes ORDER BY and result order matters (per D-11)
- `errorExpected`: optional — for error cases like cycle detection, the expected error shape (e.g., `{ kind: 'CycleError', participants: ['calc-dep-a', 'calc-dep-b'] }`)

The 10 backbone cases from 02-compilation-pipeline.md §10 are:
1. Single Calculation (derived column) — `UPPER(name) AS display_name`
2. Single Filter (equality with bind) — `node_type = 'note'`
3. Single Sort — `created_at DESC`
4. Multi-Filter AND — `node_type = 'note' AND source = 'apple-notes'`
5. Multi-Sort lexicographic — `folder_l1 ASC, name ASC`
6. Calculation + Filter combo — `UPPER(name) AS display_name` + `node_type = 'note'`
7. Aggregation with GROUP BY from view — `COUNT(*) AS row_count` with `GROUP BY node_type`
8. Window function — `RANK() OVER (PARTITION BY source ORDER BY created_at DESC) AS source_rank`
9. Cross-category filtered totals — `SUM(CASE WHEN source = ? THEN 1 ELSE 0 END) AS apple_notes_count` + `node_type = 'note'` + `GROUP BY folder_l1`
10. Dependency cycle error — A references B, B references A → CycleError

The remaining ~20 cases (per D-05 even distribution):

**~6 Formulas-focused:**

12. Three-filter AND — three chips composing

15. CASE WHEN conditional calculation — `CASE WHEN priority >= 4 THEN 'high' ELSE 'low' END AS priority_label`

**Section 3 — Test Runner Architecture (TEST-03):**

Document how the golden corpus integrates with existing infrastructure:

- **Directory structure per D-07, D-08:**
  ```
  tests/
    golden/
      fixtures/
        golden-corpus.sql       ← Extracted from §1 fixture SQL
      corpus.test.ts            ← Single parameterized test file
  ```

- **Fixture loading pattern:** A helper function `loadGoldenFixture(db: Database)` that reads the `.sql` file and runs it via `db.run()` on top of a `realDb()` instance. The helper is imported from `tests/golden/fixtures/` or defined inline in the test file. Pattern follows `realDb()` from `tests/harness/realDb.ts` — the fixture SQL runs after the schema is created by `realDb()`.

- **Test file structure:** Single `corpus.test.ts` file using Vitest `test.each()` per D-09. The test case array is a TypeScript constant of type `GoldenTestCase[]`. Each entry matches the fields from §2. Adding a new test case = adding an object to the array.

- **Assertion strategy per D-10, D-11, D-12:**
  - SQL comparison: collapse whitespace runs to single space, trim, then exact string `===`
  - Result row comparison: sort both arrays by `id` column (or first column if no id), then `expect(actual).toEqual(expected)`
  - Order-sensitive results (ORDER BY cases): skip the sort step, compare arrays directly
  - Marks annotation comparison: convert Map to sorted array of `[rowId, classes]` entries, deep-equal
  - Audits annotation comparison: convert Map to sorted array of `[rowId, annotations]` entries, deep-equal
  - Error comparison: `expect(result).toMatchObject({ kind: 'CycleError', participants: expect.arrayContaining([...]) })`
  - NO `toMatchSnapshot()` — explicit per D-12

- **Vitest configuration notes:**
  - No `@vitest-environment jsdom` annotation needed — golden corpus tests are pure SQL/data, no DOM
  - Pool: `forks` with `isolate: true` (standard WASM isolation pattern from existing test infra)
  - The test file imports from the future `FormulasProvider` (not yet implemented) — these tests are designed to drive TDD of that implementation

- **What the test exercises:** The full compile → execute → annotate pipeline:
  1. `FormulasProvider.compile(chipArrangement, viewContext)` → `{ sql, params }`
  2. `db.exec(sql, params)` → result rows
  3. `annotateMarks(rows, marksChips)` → `Map<rowId, string[]>`
  4. `annotateAudits(rows, auditsChips)` → `Map<rowId, AuditAnnotation[]>`

**Section 4 — Anti-Patching Policy (TEST-04):**

Write a standalone policy section with these exact rules (carrying forward v6.1 per FE-RG-12):

1. **Corpus assertions are immutable.** Once a test case is added to the corpus and its expected values are committed, those expected values must never be weakened, relaxed, or made more permissive to accommodate a failing implementation.
2. **If a test fails, fix the implementation or fix the spec — never weaken the assertion.** The test expected values are derived from the compilation pipeline specification (02-compilation-pipeline.md). If reality disagrees with the spec, either the implementation has a bug (fix the code) or the spec has a mistake (fix the spec and update the expected value deliberately). In no case does the assertion get weakened to make a test pass without understanding the root cause.
3. **Every bug fix adds a corpus case.** When a bug is found in the compilation pipeline, Marks annotation, or Audits annotation, the fix must include a new test case that reproduces the bug. The corpus grows monotonically.
4. **CC (Claude Code) must never weaken a corpus assertion to make a test pass.** This rule is explicit for the AI executor: if a golden test fails during implementation, the executor must diagnose the failure and fix the production code or escalate to the user. Updating expected values without understanding the root cause is prohibited.
5. **Spec changes propagate to corpus.** If the compilation pipeline spec (02-compilation-pipeline.md) is revised (e.g., a clause order change, a bind-value protocol change), the affected test case expected values are updated deliberately — this is a spec change, not an assertion weakening.

Include a grep-verifiable statement: the phrase "anti-patching" and "never weaken" must both appear in the document.
