## State Machine Visual Contract

The Notebook panel renders three mutually exclusive states. Each state has a distinct DOM layout.

### idle state

- Displays `.notebook-idle` centered panel
- Contains: muted hint text + "New Card" button (accent-colored, prominent)
- Title input is NOT rendered in idle state
- Segmented control, toolbar, and textarea are NOT rendered in idle state

Visual spec for "New Card" button:
```
display: flex
align-items: center
gap: var(--space-xs)
padding: var(--space-xs) var(--space-md)
background: var(--accent)
color: var(--bg-primary)        /* white-on-blue — sufficient contrast */
border: none
border-radius: var(--radius-md)
font-size: var(--text-sm)
font-weight: 500
cursor: pointer
min-height: 32px
transition: background-color var(--transition-fast) ease
```

Hover: `background: var(--accent-hover)`
Focus-visible: `outline: 2px solid var(--accent); outline-offset: 2px`

### buffering state

- Title input IS rendered (same `.notebook-title-input` element as editing state)
- Title input is focused immediately on entry (JS `.focus()` call)
- Title input placeholder: "Card name…"
- Segmented control, toolbar, and textarea are NOT rendered yet
- No card exists in the database — input value is held in `_bufferName` only
- A subtle visual cue distinguishes this from editing: a muted "Creating new card…" label above the title input OR the title input placeholder text alone is sufficient (use placeholder — avoids extra DOM element)
- Escape key visible affordance: none beyond standard browser behavior — Escape cancels and returns to idle

### editing state

- Identical to Phase 91 editing layout (title input + segmented control + toolbar + textarea/preview)
- No visual change from Phase 91 editing state — transition is seamless

## Interaction Contract

### "New Card" button (idle state)

- Single click: enters buffering state, focuses title input
- Keyboard: activatable with Enter or Space (standard button behavior)
- Must be the only focusable element in the idle panel (no other CTAs)

### Cmd+N shortcut

- Registered in ShortcutRegistry (existing single keydown listener)
- Guard: ShortcutRegistry already blocks TEXTAREA/INPUT elements — Cmd+N must bypass this guard because it is a meta-key shortcut (Cmd held). Verify existing metaKey shortcut behavior in ShortcutRegistry before implementation.
- When in idle state: enters buffering state (same as button click)
- When in buffering state with non-empty name: commit current name → fresh buffering state
- When in buffering state with empty name: discard current → fresh buffering state (no-op net effect)
- When in editing state: auto-commit current dirty buffer first (same as blur), then enter buffering state

### Command Palette "New Card" action

- Action label: "New Card"
- Keyboard hint displayed in palette: Cmd+N
- On activation: closes palette, opens Notebook panel if hidden, enters buffering state

### IME composition guard

- `compositionstart` → set `_isComposing = true`
- `compositionend` → set `_isComposing = false`, then re-evaluate commit/abandon
- Blur handler: if `_isComposing`, defer evaluation until `compositionend` fires
- No visual indicator needed for composing state — OS IME UI handles this

### Abandon (Escape during buffering)

- Title input is cleared
- State returns to idle
- `.notebook-idle` is restored with "New Card" button
- No animation — instant DOM swap (matches existing `_setVisible` pattern)

### Commit (blur or Cmd+S with non-whitespace name)

- INSERT fires via `createCardMutation`
- `SelectionProvider.select(newCardId)` fires immediately after INSERT resolves
- Notebook transitions to editing state with content textarea focused
- No ActionToast — the Notebook editing state IS the confirmation

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (idle state button) | "New Card" |
| Idle state hint text | "Select a card or create a new one" |
| Title input placeholder (buffering) | "Card name…" |
| Content textarea placeholder (editing) | "Write something…" (existing, unchanged) |
| Escape key cancel | no toast/message — silent return to idle |
| Cmd+N palette action label | "New Card" |
| Cmd+N palette hint | Cmd+N |

Empty state (no cards in dataset — zero-card database): not in scope for this phase. Existing empty state handling from Phase 52 (SampleDataManager) remains unchanged.

Error states: no user-visible error copy needed. Creation failures surface via existing ErrorBanner mechanism with its existing `categorizeError()` logic. No new error copy introduced.

Destructive actions: none in this phase. Card creation has no destructive path. Undo of a created card (Cmd+Z) is handled silently by the existing MutationManager pipeline — no confirmation copy needed.

## Accessibility Contract

- "New Card" button: `type="button"`, visible text label, no icon-only affordance
- Title input in buffering state: `aria-label="Card name"` (supplements placeholder for screen readers)
- State transitions (idle → buffering → editing): no ARIA live region announcement needed — focus movement provides sufficient screen reader context
- Focus management on buffering entry: `.notebook-title-input.focus()` called immediately — no delay
- Focus management on abandon (Escape): return focus to "New Card" button in idle panel
- Keyboard navigation: "New Card" button reachable via Tab; Escape exits buffering state
- All existing WCAG 2.1 AA commitments (A11Y-01..A11Y-11) remain unchanged
