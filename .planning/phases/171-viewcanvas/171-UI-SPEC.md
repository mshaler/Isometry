## Status Slot — DOM Contract

The ViewCanvas status slot renders exactly two data points. This is the primary UI artifact of Phase 171.

### DOM Structure

```
[data-slot="status"]
  └─ div.sw-status-bar
       ├─ span.sw-status-bar__item[data-stat="view-name"]   — e.g. "SuperGrid"
       ├─ span.sw-status-bar__sep                            — "·"
       └─ span.sw-status-bar__item[data-stat="card-count"]  — e.g. "1,243 cards"
```

- Reuses `.sw-status-bar` CSS class defined in `src/styles/superwidget.css` (line 206–215).
- Idempotent render: skip DOM creation if `.sw-status-bar` already present (same guard as `renderStatusSlot()`).
- Update function signature matches ExplorerCanvas pattern: pure DOM mutation, no return value.

### CSS Classes Reused (no new rules needed)

| Class | Defined in | Purpose |
|-------|-----------|---------|
| `.sw-status-bar` | superwidget.css:206 | Flex row, 4px gap, 4px/8px padding, 11px font, muted color |
| `.sw-status-bar__item` | superwidget.css (implied by `__sep` pattern) | Individual stat span |
| `.sw-status-bar__sep` | superwidget.css:217 | Separator dot, user-select: none |

## Interaction Contract

### Sidecar Visibility

- Driven by `registry.defaultExplorerId` for the active view (D-05, D-06).
- ViewCanvas calls `onSidecarChange(explorerId: string | null)` after each `switchTo()`.
- Views with a `defaultExplorerId` → sidecar visible.
- Views without → `null` passed → sidecar hidden.
- No animation spec required; SuperWidget handles the transition (existing behavior).

### View Switching

- `onProjectionChange(proj)` → cast `proj.activeTabId` to `ViewType` → call `ViewManager.switchTo()`.
- Invalid tab ID: validation error, no silent fallback (D-08).
- Destroy-before-mount ordering enforced for rapid switching (D-11, < 500ms guarantee).

### Status Slot Update Timing

- Status slot spans update in the ViewManager render callback (D-03).
- Not on a timer. Not on provider change directly.
- Update function is called once per render cycle completion.
