  <name>Task 2: Configure Vite and Vitest for sql.js WASM</name>
  <files>vite.config.ts, vitest.config.ts, tests/setup/wasm-init.ts</files>
  <action>
1. Create `vite.config.ts` per research Pattern 2:
```typescript
import { defineConfig } from 'vite';
import { viteStaticCopy } from 'vite-plugin-static-copy';

export default defineConfig({
  plugins: [
    viteStaticCopy({
      targets: [
        {
          src: 'src/assets/sql-wasm-fts5.wasm',
          dest: 'assets'
        }
      ]
    })
  ],
  worker: {
    format: 'es',
  },
  optimizeDeps: {
    exclude: ['sql.js'],  // CRITICAL: prevents esbuild from stripping WASM loading code
  },
  build: {
    target: 'es2022',
    assetsInlineLimit: 0,  // Never inline WASM (breaks WKWebView + streaming compile)
  },
});
```

CRITICAL configuration notes (from research anti-patterns):
- `optimizeDeps: { exclude: ['sql.js'] }` -- Without this, esbuild pre-bundles sql.js and strips WASM loading code entirely.
- `assetsInlineLimit: 0` -- Without this, Vite may inline small assets as base64 data URIs, breaking WASM streaming compilation.
- `worker: { format: 'es' }` -- Required for WKWebView ES module workers.

2. Create `vitest.config.ts` per research Pattern 4:
```typescript
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    environment: 'node',   // NOT jsdom -- WASM runs in Node directly
    pool: 'forks',         // Full process isolation -- prevents WASM state leakage
    isolate: true,         // Each test file gets its own process
    globalSetup: './tests/setup/wasm-init.ts',
    globals: true,
    testTimeout: 10000,    // WASM init can take 1-2 seconds on cold start
    coverage: {
      provider: 'v8',
      reporter: ['text', 'lcov'],
      include: ['src/**/*.ts'],
    },
  },
});
```

CRITICAL: Do NOT use `pool: 'threads'` -- sql.js WASM state leaks between test files in threads pool. Use `pool: 'forks'` with `isolate: true`.

3. Create `tests/setup/wasm-init.ts`:
```typescript
import { fileURLToPath } from 'url';
import { resolve, dirname } from 'path';
import { existsSync } from 'fs';

export async function setup() {
  // First try custom FTS5 build in src/assets/
  const customWasmPath = resolve(
    dirname(fileURLToPath(import.meta.url)),
    '../../src/assets/sql-wasm-fts5.wasm'
  );

  if (existsSync(customWasmPath)) {
    process.env.SQL_WASM_PATH = customWasmPath;
    return;
  }

  // Fallback to default sql.js dist (no FTS5, but tests can still run for config validation)
  const defaultWasmPath = resolve(
    dirname(fileURLToPath(import.meta.url)),
    '../../node_modules/sql.js/dist/sql-wasm.wasm'
  );

  if (existsSync(defaultWasmPath)) {
    process.env.SQL_WASM_PATH = defaultWasmPath;
    return;
  }

  throw new Error('No sql.js WASM file found. Run the custom FTS5 build first or ensure sql.js is installed.');
}
```

This setup provides the WASM path via environment variable to all test files, resolving the Vitest WASM path resolution issue (Pitfall 10).
