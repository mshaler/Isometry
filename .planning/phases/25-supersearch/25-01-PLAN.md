**4. Compute matchedCardIds in handleSuperGridQuery (supergrid.handler.ts):**

After the existing `cells` construction loop, add:

```typescript
// Phase 25 SRCH-04 — compute matchedCardIds per cell when search is active
const trimmedSearch = payload.searchTerm?.trim() ?? '';
if (trimmedSearch) {
  // Secondary FTS5 query: get all card IDs that match the search term
  const ftsResults = db.exec(
    'SELECT c.id FROM cards_fts JOIN cards c ON c.rowid = cards_fts.rowid WHERE cards_fts MATCH ? AND c.deleted_at IS NULL',
    [trimmedSearch]
  );
  const matchedSet = new Set<string>();
  if (ftsResults.length > 0 && ftsResults[0]!.values) {
    for (const row of ftsResults[0]!.values) {
      if (typeof row[0] === 'string') matchedSet.add(row[0]);
    }
  }

  // Annotate each cell with which of its card_ids matched FTS
  for (const cell of cells) {
    cell['matchedCardIds'] = cell.card_ids.filter(id => matchedSet.has(id));
  }

  return { cells, searchTerms: [trimmedSearch] };
}
```

The existing `return { cells };` handles the no-search case (no matchedCardIds, no searchTerms).

**Critical constraints:**
- FTS5 MATCH must NEVER receive empty string — guard with `.trim()` check (Pitfall 1 from RESEARCH.md)
- `matchedCardIds` uses `cell['matchedCardIds']` bracket notation — fits CellDatum's `[key: string]: unknown` index signature
- Search params MUST be appended AFTER filter params in the params array (positional SQL parameters)
- The secondary FTS query in the handler uses `db.exec()` (not `db.prepare()`) because it returns columnar results which are simpler for a flat `id` extraction
