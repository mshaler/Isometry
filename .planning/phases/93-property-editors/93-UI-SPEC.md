## Spacing Scale

Existing scale from `design-tokens.css` (theme-independent `:root` block). All values are multiples of 4.

| Token | Value | Usage |
|-------|-------|-------|
| `--space-xs` | 4px | Icon gaps, chip gaps, inline padding, field row vertical padding |
| `--space-sm` | 8px | Compact element spacing, field row horizontal padding |
| `--space-md` | 12px | Section internal padding, label-to-input gap |
| `--space-lg` | 16px | Section header padding, group separation |
| `--space-xl` | 24px | Panel-level padding, group-to-group gap |

Spacing Scale Exceptions:

- `--space-md: 12px` — pre-existing project token defined in `design-tokens.css` line 113 (theme-independent `:root` block). Not introduced by this phase. 12px is not in the standard 8-point set `{4, 8, 16, 24, 32, 48, 64}` but is an established live token used across the codebase.
- Tag chip pill horizontal padding: 6px — sub-grid alignment for compact pill shape. Not a structural spacing value; it is a visual padding applied inside the pill element where 4px is too tight and 8px is too wide for a label chip. This is the single exception; all other spacing uses the token scale.
- Minimum touch target for toggle/checkbox: 20px height (compact panel context, not a touch device).
- Tag chip remove button: 16px × 16px clickable area inside the pill.

Source: `design-tokens.css` `:root` spacing block — pre-populated, not modified.

## Typography

All values from `design-tokens.css` `:root` typography scale. Executor must not introduce new sizes.

| Role | Token | Size | Weight | Line Height |
|------|-------|------|--------|-------------|
| Body / field value | `--text-sm` | 11px | 400 (regular) | 1.5 |
| Field label | `--text-xs` | 10px | 600 (semibold) | 1.4 |
| Section heading | `--text-sm` | 11px | 600 (semibold) | 1.4 |
| Card type select / date input | `--text-base` | 13px | 400 (regular) | 1.4 |

Declared sizes used: 10px (`--text-xs`), 11px (`--text-sm`), 13px (`--text-base`).
Declared weights used: 400 (regular), 600 (semibold).

Rationale: The property panel is a dense, compact sidebar panel. The existing PropertiesExplorer uses `--text-xs` labels and `--text-xs` body — this phase mirrors that density. The 10px (`--text-xs`) and 11px (`--text-sm`) sizes are only 1px apart in absolute terms, but their visual differentiation is carried entirely by weight: labels are always semibold (600) and values are always regular (400). This weight contrast — not size contrast — is the intended signal, matching the existing PropertiesExplorer label/value pattern. The card_type `<select>` gets `--text-base` (13px) because native `<select>` renders at system size by default and 11px causes visual undersizing on macOS.

Source: `design-tokens.css` — pre-populated.

## Color

All tokens from `design-tokens.css`. The 60/30/10 allocation applies to the CardEditorPanel surface area within the Workbench sidebar. The focal point of the CardEditorPanel surface is the active input field: the accent focus ring (`outline: 2px solid var(--accent)`) draws the eye to the currently edited control within the otherwise uniform `--bg-primary` field of the panel.

| Role | Token | Value (dark default) | Usage |
|------|-------|---------------------|-------|
| Dominant (60%) | `--bg-primary` | `#1a1a2e` | CardEditorPanel background, input backgrounds |
| Secondary (30%) | `--bg-surface` | `#252540` | Section group headers, tag chip pill backgrounds, field row hover |
| Accent (10%) | `--accent` | `#4a9eff` | Active input focus ring, selected toggle state, tag chip border on focus |
| Destructive | `--danger` | `#ff4a4a` | Tag pill remove button hover, validation rejection flash |

Accent reserved for:
1. `outline: 2px solid var(--accent)` on `:focus-visible` for every input, select, toggle, and chip remove button
2. Active `<input type="checkbox">` / toggle fill via `accent-color: var(--accent)`
3. Focused tag chip border: `border-color: var(--accent-border)`
4. Date/number input active border: `border-color: var(--accent)` on `:focus`

Destructive color reserved for:
- Tag pill remove ("×") button background on `:hover` — `background: var(--danger-bg)`, icon color `var(--danger)`
- Number input rejection state (non-numeric value entered and rejected): brief 150ms `border-color: var(--danger)` flash, no persistent error state

## Field Grouping

26 card fields organized into 5 collapsible groups. All groups start expanded. Success criterion 5 (all fields reachable without scrolling to a hidden section) is met by keeping all groups expanded by default and making the panel container scrollable.

| Group | Fields | Input Type |
|-------|--------|------------|
| **Identity** | `card_type`, `name` (read-only in this panel — edited via title input), `summary` | `card_type` → select; `summary` → text input |
| **Organization** | `folder`, `status`, `tags`, `is_collective`, `priority`, `sort_order` | text, text, chip editor, toggle, number, number |

## Interaction Contracts

### Shadow-Buffer Pattern (PROP-08)

Each field follows the same pattern established in Phase 91 for title/content:

1. On card load: snapshot all 26 field values into a local buffer object
2. On input change: update the in-memory buffer only (no mutation)
3. On commit (blur from the specific field): call `updateCardMutation(card.id, snapshot, { [field]: coercedValue })` — one step per field per editing session
4. Before the next card loads: any dirty buffer fields are committed (same blur path)

### Null-Safe Coercion (PROP-09)

Coercion rules applied in the utility function before every `updateCardMutation` call:

| Field type | Empty / invalid input | Coercion result |
|-----------|----------------------|-----------------|
| Nullable text (`summary`, `folder`, `status`, `url`, `mime_type`, `source`, `source_id`, `source_url`, `location_name`) | `""` (empty string) | `null` |
| Non-nullable text (`name`) | `""` | blocked — title input prevents commit of empty name |
| Number (`priority`, `sort_order`) | non-numeric string | rejected by `<input type="number">` — browser prevents input |
| Nullable number (`latitude`, `longitude`) | `""` | `null` |
| Date (`due_at`, `completed_at`, `event_start`, `event_end`) | `""` | `null`; non-empty value → ISO 8601 string (`YYYY-MM-DD`) |
| Boolean (`is_collective`) | — | `true` / `false` from checkbox `.checked` |
| `card_type` | — | always one of `'note' \| 'task' \| 'event' \| 'resource' \| 'person'` |
| `tags` | — | `string[]`, empty array is valid (not null) |

### card_type Select (PROP-07)

The `<select>` for `card_type` uses the `updateCard` path extended in Phase 93 (PROP-07 restriction lifted). Commits on `change` event (not blur), because select changes are discrete choices. A single `updateCardMutation` step is created immediately on selection.

Options rendered as:
```
<option value="note">Note</option>
<option value="task">Task</option>
<option value="event">Event</option>
<option value="resource">Resource</option>
<option value="person">Person</option>
```

### Tag Chip Autocomplete (PROP-06)

Autocomplete sources from a `SELECT DISTINCT json_each.value FROM cards, json_each(cards.tags)` query run once when the panel binds to a card. Results are rendered as a `<datalist>` attached to the `.cpf-tag-input` via `list` attribute. No custom dropdown — native datalist is sufficient for this phase.

### Undo Safety

Each field commit is one `updateCardMutation` step. Cmd+Z undoes exactly that one field. Unrelated fields in the same card are not affected because the mutation carries a precise `{field: newValue}` patch with the before-snapshot stored by `updateCardMutation`.

## Accessibility Contract

- Every `.cpf-row__label` is associated to its control via `for`/`id` pairing. Label text matches the field name in sentence case.
- The tag chip editor's remove buttons each carry `aria-label="Remove tag {tagValue}"`.
- The `.cpf-group__header` button carries `aria-expanded="true|false"` and `aria-controls="{group-body-id}"`.
- Focus order follows DOM order: Identity → Organization → Time → Location → Source. Tab moves field-by-field.
- The tag chip editor supports keyboard-only addition and removal (Enter adds, Backspace removes last chip, Tab commits pending text).
- `<input type="date">` uses native date picker — no custom calendar widget.
- `<input type="checkbox">` for `is_collective` uses native toggle — `accent-color: var(--accent)` for brand alignment.
