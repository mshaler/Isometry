<feature>
  <name>Feature A: FilterProvider.setRangeFilter()</name>
  <files>src/providers/types.ts, src/providers/FilterProvider.ts, tests/providers/FilterProvider.test.ts</files>
  <behavior>
    - setRangeFilter('priority', 1, 5) stores a range filter for 'priority' with min=1, max=5
    - compile() produces "priority >= ? AND priority <= ?" with params [1, 5]
    - Calling setRangeFilter('priority', 2, 8) REPLACES the previous range (not appends)
    - setRangeFilter with min only (max=null): produces "priority >= ?" only
    - setRangeFilter with max only (min=null): produces "priority <= ?" only
    - clearRangeFilter('priority') removes the range without affecting other filters or axis filters
    - clearFilters() clears range filters alongside regular and axis filters
    - resetToDefaults() clears range filters
    - hasActiveFilters() returns true when range filters exist
    - Range filters compile AFTER axis filters, BEFORE FTS (consistent ordering)
    - toJSON() includes rangeFilters as Record<string, {min: unknown, max: unknown}>
    - fromJSON round-trip preserves range filter state
    - setState() with missing rangeFilters defaults to empty (backward compat)
    - setState() validates rangeFilters shape (must be object with min/max per key)
    - setRangeFilter throws SQL safety violation for invalid field
    - clearRangeFilter throws SQL safety violation for invalid field
    - setRangeFilter fires subscriber notification via _scheduleNotify
    - clearRangeFilter fires subscriber notification
    - hasRangeFilter('priority') returns true when set, false when not
    - Two rapid setRangeFilter calls produce only ONE notification (batching)
  </behavior>
  <implementation>
    1. Add RangeFilter interface to src/providers/types.ts:
       ```typescript
       export interface RangeFilter { min: unknown; max: unknown; }
       ```
    2. Add optional rangeFilters to FilterState interface in FilterProvider.ts:
       ```typescript
       rangeFilters?: Record<string, RangeFilter>;
       ```
    3. Add private _rangeFilters Map<string, RangeFilter> field to FilterProvider class.
    4. Add setRangeFilter(field: string, min: unknown, max: unknown) method:
       - validateFilterField(field) for SQL safety
       - If both min and max are null/undefined, delete the entry (equivalent to clear)
       - Otherwise store { min, max } in the Map
       - Call _scheduleNotify()
    5. Add clearRangeFilter(field: string) method:
       - validateFilterField(field) for SQL safety
       - Delete from Map
       - Call _scheduleNotify()
    6. Add hasRangeFilter(field: string) method: returns _rangeFilters.has(field).
    7. Update compile() to iterate _rangeFilters after axis filters and before FTS:
       - For each entry, emit "field >= ?" if min is not null, "field <= ?" if max is not null
       - Push min/max to params accordingly
       - validateFilterField(field) for runtime safety (JSON-restored state)
    8. Update hasActiveFilters() to include _rangeFilters.size > 0.
    9. Update clearFilters() to call _rangeFilters.clear().
    10. Update resetToDefaults() to call _rangeFilters.clear().
    11. Update toJSON() to include rangeFilters: Object.fromEntries(_rangeFilters).
    12. Update isFilterState() guard to validate optional rangeFilters shape.
    13. Update setState() to restore _rangeFilters from state.rangeFilters (with backward compat default).
  </implementation>
</feature>

<feature>
  <name>Feature B: histogram:query Worker handler</name>
  <files>src/worker/protocol.ts, src/worker/worker.ts, src/worker/handlers/histogram.handler.ts, src/worker/handlers/histogram.handler.test.ts</files>
  <behavior>
    - histogram:query with field='priority', bins=10 returns array of { binStart, binEnd, count }
    - Numeric fields: SQL CASE expressions create equal-width bins from MIN to MAX
    - Date fields: strftime-based bucketing (e.g., by month '%Y-%m') returns { binStart, binEnd, count }
    - Respects WHERE clause from FilterProvider.compile() (only bins visible data)
    - Field is validated via validateFilterField() (SQL safety)
    - fieldType='numeric' triggers CASE-based binning
    - fieldType='date' triggers strftime-based monthly bucketing
    - Empty data (no rows matching WHERE) returns empty array []
    - All-NULL column returns empty array []
    - Single distinct value returns one bin with count
    - bins parameter defaults to 10 if not provided
    - WHERE clause defaults to 'deleted_at IS NULL' if not provided
  </behavior>
  <implementation>
    1. Add 'histogram:query' to WorkerRequestType union in protocol.ts.
    2. Add WorkerPayloads['histogram:query'] type:
       ```typescript
       'histogram:query': {
         field: string;        // Column to bin (validated against filter allowlist)
         fieldType: 'numeric' | 'date';  // Determines binning strategy
         bins: number;         // Number of bins (default 10)
         where: string;        // From FilterProvider.compile()
         params: unknown[];    // From FilterProvider.compile()
       };
       ```
    3. Add WorkerResponses['histogram:query'] type:
       ```typescript
       'histogram:query': {
         bins: Array<{ binStart: number | string; binEnd: number | string; count: number }>;
       };
       ```
    4. Create src/worker/handlers/histogram.handler.ts with handleHistogramQuery(db, payload):
       - validateFilterField(payload.field) for SQL safety
       - For numeric fields:
         a. Query MIN(field) and MAX(field) from cards WHERE {where} AND field IS NOT NULL
         b. If no rows or min===max, return single bin or empty array
         c. Compute bin width = (max - min) / bins
         d. Build SQL with CASE WHEN expressions assigning each row to a bin index
         e. GROUP BY bin index, COUNT(*) per bin
         f. Map results to { binStart, binEnd, count } array
       - For date fields:
         a. Use strftime('%Y-%m', field) to bucket by month
         b. SELECT strftime('%Y-%m', field) AS bucket, COUNT(*) AS count ... GROUP BY bucket ORDER BY bucket ASC
         c. Map results to { binStart: bucket, binEnd: bucket, count } array
    5. Wire handler in worker.ts: add case 'histogram:query' dispatching to handleHistogramQuery.
    6. Import handleHistogramQuery in worker.ts.

    IMPORTANT: Use db.prepare<T>(sql) + stmt.all(...params) + stmt.free() pattern matching chart.handler.ts exactly. Never use db.exec() for SELECT queries.
  </implementation>
</feature>
