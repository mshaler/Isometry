ALL of these must move into registerCatalog().

The rerender callback problem: superstack.collapse and supercalc.config need a rerender callback.
Solution from CONTEXT.md: use registry.onChange() mechanism. PivotTable already subscribes to
registry.onChange() for re-render. So collapse/config plugins should call a notify function that
triggers registry.onChange(). The registerCatalog function receives the registry, so it can create
a notify helper: `const notify = () => { (registry as any)._notify(); }` -- BUT that uses a private
method. Better approach: registerCatalog creates shared state with a `rerender` callback placeholder,
then the factories capture it. The HarnessShell's registry.onChange() subscription in mount() already
calls pivotTable.rerender(). So plugins just need to trigger a registry notification.

Actual solution: The PluginRegistry.setFactory() method already handles hot-swap. When a plugin
calls onCollapseToggle(), we need to trigger the registry's onChange listeners. The cleanest way:
pass a `notify` function that iterates `registry._listeners`. But _listeners is private.

Pragmatic solution per CONTEXT.md: The registerCatalog function already has access to the registry
parameter. Create a wrapper: `const triggerRerender = () => { registry.enable('superstack.collapse'); registry.disable('superstack.collapse'); }` -- No, that's ugly.

Best solution: Add a public `notify()` method to PluginRegistry, or use the existing pattern where
HarnessShell.mount() subscribes to onChange and rerenders. The collapse plugin just needs to call
a function that the shell provides. Since we're moving ALL wiring into registerCatalog, we need
registerCatalog to accept an optional rerender callback OR we add a thin `registry.triggerChange()`
public method.

The CONTEXT.md says: "plugins trigger re-render through registry's onChange mechanism -- PivotTable
already subscribes to registry.onChange(), so no new API needed." This implies we should add a small
public triggerChange() or notifyListeners() method to PluginRegistry.
