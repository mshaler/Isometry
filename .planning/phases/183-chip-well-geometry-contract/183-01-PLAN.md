**Section 1 — Intent (per GEOM-01, UI-SPEC §Document Structure Contract row 1):**
Two paragraphs:
1. What the chip well IS: a layout primitive for typed configuration. It hosts draggable chip tokens organized into typed categories. Its geometry contract specifies spatial behavior — coordinate system, drag affordances, drop-zone semantics, reflow rules — independent of which explorer uses it or what the chips mean.
2. What the chip well is NOT: not a data view, not PAFV-bound. It produces typed configuration consumed by view explorers; views own PAFV binding. Not a component spec or style guide.

**Section 2 — Coordinate system (per GEOM-02, UI-SPEC §Document Structure Contract row 2):**
- Authority: the well container element owns layout; chips are subordinate
- Units: chip token — variable width (content-sized label + padding), fixed height 24px at `--size-card-mini`. Chip vertical padding: 2px (`chip-v-pad` named codebase structural value). Chip horizontal padding: `--space-sm` (8px). Chip-to-chip gap: `--space-xs` (4px). Well internal vertical padding: `--space-md` (12px). Between-well vertical gap: `--space-lg` (16px). Panel section break: `--space-xl` (24px).
- Origin: top-left CSS coordinate convention. Positive-x rightward, positive-y downward.
- Axes: chips flow left-to-right within a well row (inline axis), wrapping to new rows when horizontal space exhausts. Wells stack top-to-bottom within the explorer panel (block axis).
- Well header: `--text-xs` (10px), 600 weight, uppercase, 0.5px letter-spacing, `--space-sm` bottom margin
- Chip label: `--text-xs` (10px), 400 weight, line-height 1.4

**Section 3 — PAFV binding (per GEOM-01, FE-RG-13):**

**Section 4 — Data binding (per UI-SPEC §Document Structure Contract row 4):**
- Input contract: an ordered list of chips per well. Each chip carries a type signature and a DSL fragment. The list order is user-controlled and semantically meaningful (e.g., sort priority for ordering chips, dependency order for expression chips).
- Output contract: same shape, mutated by user action (reorder, add, remove). Published via `ChipWellOutputContract` (named interface only — Phase 184 owns implementation).
- Cardinality expectations: Empty (0 chips — empty state per §7), Sparse (1-5 chips — typical single-well), Typical (5-20 chips — multi-category active configuration), Dense (20+ chips — heavy configuration, wrapping expected). No virtualization needed — chip wells are control surfaces with bounded cardinality.
- No aggregate handling (chip wells are not data views).

**Section 5 — Invariants (per UI-SPEC §Document Structure Contract row 5):**
At minimum these invariants:
- Chip order within a well is user-controlled and semantically meaningful; the system never silently reorders chips
- Chips never silently change category (move between wells without user action)
- Drop-zone hit testing is deterministic — given the same pointer position and well layout, the same drop target is identified
- Cross-well drag copies by default; modifier key (platform-standard: Alt/Option) triggers move; never rejects if type signatures are compatible (STATE.md locked decision)
- A chip's visual representation is independent of which explorer hosts the well — the geometry contract is generic (FE-RG-14)
- Empty wells are valid and render an empty state; they do not collapse or hide

**Section 6 — Degenerate and edge cases (per UI-SPEC §Document Structure Contract row 6, D-05, D-06, D-07):**
Table with at minimum these cases:
| Case | Expected behavior |
| Empty well (0 chips) | Empty state: "Drop chips here" heading, "Drag a chip from another well or the Formula Card library to add it." body (exact copy from UI-SPEC Copywriting Contract) |
| Single chip | Well renders normally with one chip; no special layout adjustments |
| Many chips (overflow) | Chips wrap to additional rows within the well; no hard chip limit (D-05); if wrapped content exceeds panel height, panel scrolls vertically |
| Type-incompatible drop | Well enters drag-target-invalid state (§7); chip returns to source on drop; live-region announces rejection per UI-SPEC copy |
| Drag abort mid-flight | Chip returns to original position; no persistent visual change (D-07); semantic description only, no animation timing (D-02) |
| Cross-explorer drag | Same rules as cross-well: copy by default, modifier for move, never reject if type signatures match |
| Touch drag abort (finger lift outside any well) | Same as pointer abort — chip returns to origin |
| Well with long chip labels | Chip width grows with content up to a max-width constraint (implementation-defined); overflowing text truncates with ellipsis |

**Section 7 — States (per GEOM-03, D-01, D-02, UI-SPEC §Color > Drag state color assignments):**
All 6 states with semantic visual treatment (function not chrome per D-01) and pointer event behavior. No CSS properties, no animation timing (D-02). Use token names from UI-SPEC:

1. **default** — Chip: `--bg-surface` background, `--border-subtle` border, `--text-primary` label. Well: neutral background. All pointer events active.
2. **drag-source-active** — Chip being dragged: reduced opacity (0.6), `--bg-surface` background. Source well: `--border-muted` border. Drag chip follows pointer. Other wells evaluate drop compatibility.
3. **drag-target-valid** — Target well: `--accent-bg` background fill, `--accent-border` border, `--accent` label on well header. Drop indicator shows insertion point. `pointerup` commits the drop.
4. **drag-target-invalid** — Target well: `--danger-bg` background fill, `--danger-border` border. `pointerup` rejects the drop. Live-region announces rejection.
5. **drop-rejected** — Chip snaps back to origin; no persistent visual change; brief `--danger` border flash (end-state only per D-02). Live-region announces: "[chip label] cannot be dropped here. Returned to [source well name]."
6. **promotion-prompt** — `<dialog>` overlay appears with heading "Save as Formula", body "Create a reusable Formula Card from this chip arrangement.", primary CTA "Save Formula", cancel "Cancel". Chip in well shows `--accent-border` ring. Rest of panel dims via `--overlay-bg`. Dialog: `aria-labelledby` pointing to heading.

Include state machine diagram (mermaid):
```
[*] --> default
default --> drag-source-active: pointerdown on chip (mouse) or long-press 500ms (touch)
drag-source-active --> drag-target-valid: pointermove over compatible well
drag-source-active --> drag-target-invalid: pointermove over incompatible well
drag-source-active --> default: pointerup outside any well (abort) or Escape
drag-target-valid --> default: pointerup (commit drop)
drag-target-invalid --> drop-rejected: pointerup (reject)
drop-rejected --> default: chip returns to origin
default --> promotion-prompt: user triggers "Save as Formula"
promotion-prompt --> default: dialog confirm or cancel
```

Reflow rules: chip wells are embedded within explorer panels. Panel width determines chip wrapping. No breakpoint-specific behavior — the wrapping model handles all widths. Wells do not collapse or reflow into a different arrangement at narrow widths.

Empty state: well container renders with `--bg-surface` background, centered text: heading "Drop chips here" (`--text-sm`, 600 weight), body below it (`--text-sm`, 400 weight).
