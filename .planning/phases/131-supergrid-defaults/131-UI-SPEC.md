## Component Inventory

Phase 131 introduces one new UI element and reuses three existing patterns.

### New: "Reset to defaults" button in ProjectionExplorer footer

- **Class:** `projection-explorer__reset-btn` (new class, following `properties-explorer__reset-btn` precedent)
- **Appearance:** Match `properties-explorer__footer-btn` exactly — font-size `var(--text-xs)`, padding `2px 8px`, border `1px solid var(--border-muted)`, border-radius `var(--radius-sm)`, background `var(--bg-surface)`, color `var(--text-secondary)`
- **Placement:** Footer area of ProjectionExplorer panel, right-aligned within the footer flex row
- **Visibility:** Hidden when no per-dataset override exists (`view:defaults:applied:{datasetId}` flag set, no ui_state overrides). Visible as soon as any manual axis change is made.
- **Label:** `Reset to defaults`
- **data-testid:** `projection-explorer-reset-btn`

### Reused: AppDialog confirmation for reset action

- **Variant:** `confirm` (not `danger` — reset is reversible/non-destructive)
- **Title:** `Reset to Defaults`
- **Message:** `Restore source-type default axes and layout for this dataset?`
- **Confirm label:** `Reset`
- **Cancel label:** `Cancel`

Source pattern: `src/ui/PropertiesExplorer.ts` lines 525–536, `src/styles/app-dialog.css`.

### Reused: SuperGrid grid body (no visual changes)

The SuperGrid grid itself has no visual changes in this phase. Axis/sort/density/calc values are wired at the provider level. The grid re-renders automatically via the existing D3 data join.

### Reused: existing footer-btn pattern

No new CSS class needed for the button shape — use existing `properties-explorer__footer-btn` style rules or duplicate them under `.projection-explorer__footer-btn` for namespace separation. Match exactly.

## Interaction States

### "Reset to defaults" button

| State | Appearance |
|-------|-----------|
| Default (overrides present) | Visible, `var(--bg-surface)` background, `var(--text-secondary)` text |
| Hover | `var(--cell-hover)` background overlay (rgba 4–8% opacity) |
| Focus-visible | `2px solid var(--accent)` outline, `outline-offset: -2px` |
| Active/pressed | No separate style needed (fast action) |
| Hidden (no overrides) | `display: none` — only shown when per-dataset overrides exist |

### First-import auto-apply (no user-visible interaction)

The default axis apply on first import is silent — no toast, no animation, no explicit notification. The grid simply renders with the correct axes. This is intentional: the grid always shows something meaningful, so no banner or announcement is needed.

### Fallback state (expected column missing)

When a default column is absent, the best-available fallback is applied silently. No error state, no toast. The grid renders with whatever valid column was found. This is a silent degradation — same visual result as the happy path.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Reset button label | `Reset to defaults` |
| AppDialog title | `Reset to Defaults` |
| AppDialog body | `Restore source-type default axes and layout for this dataset?` |
| AppDialog confirm | `Reset` |
| AppDialog cancel | `Cancel` |
| Empty state (no axis columns available) | Not applicable — SchemaProvider fallback always yields at least one valid column; if truly zero columns, existing SuperGrid empty state handles it |
| Error state (schema validation fails) | No user-facing error — SGDF-02/SGDF-03 mandate silent fallback, not error display |

Notes on copy decisions:
- "Reset to defaults" (not "Reset defaults") — preposition "to" clarifies direction of the action.
- Dialog body uses "Restore" not "Reset" to avoid repeating the title verb and to convey that something is being brought back, not erased.
- "source-type default axes and layout" — specific enough to tell the user what will change (axes + layout), what determines the defaults (source type), without enumerating every field.
