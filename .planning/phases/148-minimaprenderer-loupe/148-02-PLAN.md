**Part A: Add loupe overlay to MinimapRenderer.ts**

Add exported function `renderLoupe(thumbEl: HTMLElement, viewportRect: { x: number; y: number; w: number; h: number }): void`:
- `viewportRect` values are in 0-1 normalized coordinates (0,0 = top-left of full canvas, 1,1 = bottom-right)
- Find the `.minimap-svg` inside thumbEl; if missing, return early
- Remove any existing `.minimap-loupe` group from the SVG
- Create a `<g class="minimap-loupe">` element
- Convert normalized coords to SVG pixels: `px = x * 96`, `py = y * 48`, `pw = w * 96`, `ph = h * 48` (clamped to SVG bounds)
- Create 4 dimming `<rect>` elements (inverted dimming pattern per D-05):
  - Top: x=0 y=0 width=96 height=py
  - Bottom: x=0 y=py+ph width=96 height=48-(py+ph)
  - Left: x=0 y=py width=px height=ph
  - Right: x=px+pw y=py width=96-(px+pw) height=ph
  - All four: fill=`var(--overlay-bg)` (which is `rgba(0,0,0,0.5)` from design tokens)
- Create viewport outline `<rect>`: x=px y=py width=pw height=ph, fill="none", stroke=`var(--accent)`, stroke-width=2, rx=1
- Append group to SVG
- Set SVG cursor: `svg.style.cursor = 'crosshair'`

Add pointer interaction setup as a separate function `attachLoupeInteraction(thumbEl: HTMLElement, onNavigate: (normX: number, normY: number) => void): () => void`:
- Find `.minimap-svg` in thumbEl; if missing, return no-op cleanup
- On `pointerdown`: call `svg.setPointerCapture(e.pointerId)`, set `svg.style.cursor = 'grabbing'`, set dragging flag
- On `pointermove` (only while dragging): compute normalized coords from pointer position relative to SVG bounds (`e.offsetX / 96`, `e.offsetY / 48`), clamp 0-1, call `onNavigate(normX, normY)`
- On `pointerup`: release pointer capture, set `svg.style.cursor = 'crosshair'`, clear dragging flag
- On `click` (when not preceded by drag > 3px movement): compute normalized coords, call `onNavigate(normX, normY)`
- Return a cleanup function that removes all three event listeners

2. Wire the navigate callback — this is the critical signal path from loupe interaction to actual view scrolling (per MMAP-03, D-04):
   ```typescript
   dockNav.setNavigateCallback((normX, normY) => {
     // The view container is the scrollable element that ViewManager manages
     const container = viewManager.container;  // or however the container is accessed
     // Scroll the container to the normalized position
     // normX/normY are 0-1 representing position within the full content area
     const maxScrollLeft = container.scrollWidth - container.clientWidth;
     const maxScrollTop = container.scrollHeight - container.clientHeight;
     container.scrollTo({
       left: normX * maxScrollLeft,
       top: normY * maxScrollTop,
       behavior: 'instant',  // instant for drag-to-pan responsiveness
     });
   });
   ```
   NOTE: Read ViewManager.ts to determine how the container is exposed. The `container` field is private — either use a public getter if one exists, or read main.ts to find the container element that was passed to the ViewManager constructor and reference it directly (it's the same HTMLElement). The key insight: the ViewManager's container IS the scrollable viewport for DOM-based views (SuperGrid, list, etc.). For SVG views (network, tree) that use D3 zoom transforms rather than native scroll, the `scrollWidth === clientWidth` so the normalized scroll will be a no-op — which is acceptable for v1 since those views fit their content to the viewport. A future phase can add per-view-type navigation if needed.

3. Wire re-render triggers — subscribe to coordinator changes and view switches:
   ```typescript
   coordinator.subscribe(() => {
     dockNav.requestThumbnailUpdate();
   });
   ```
   This piggybacks on the existing coordinator subscription pattern. The debounce inside `requestThumbnailUpdate()` prevents flooding.

4. Chain the onViewSwitch callback (do not replace existing callback if one is set):

   ```typescript
   viewManager.onViewSwitch = (viewType) => {
     existingOnSwitch?.(viewType);
     dockNav.requestThumbnailUpdate();
   };
   ```
