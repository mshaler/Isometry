    **1. Add private flag to ViewManager class:**

    ```typescript

    ```

    **2. Wrap both paths in switchTo() with the flag (per D-03: silent drop):**

    At the very top of `switchTo()`, before any work:
    ```typescript
    this._isSwitching = true;
    ```

    At the very bottom of `switchTo()`, after all work (both morph and crossfade paths), in a `finally`-like pattern. Since switchTo has two code paths (morph and crossfade), the cleanest approach is to wrap the entire method body in try/finally:

    ```typescript
    async switchTo(viewType: ViewType, createView: (type: ViewType) => IView): Promise<void> {
      this._isSwitching = true;
      try {
        // ... existing morph/crossfade logic unchanged ...
      } finally {
        this._isSwitching = false;
      }
    }
    ```

    **3. Guard the coordinator subscription callback:**

    In BOTH the morph path (line ~255) and the crossfade path (line ~313), the coordinator subscription callback is:
    ```typescript
    this.coordinatorUnsub = this.coordinator.subscribe(() => {
      void this._fetchAndRender();
    });
    ```

    Change both to:
    ```typescript
    this.coordinatorUnsub = this.coordinator.subscribe(() => {
      if (this._isSwitching) return;
      void this._fetchAndRender();
    });
    ```

    Note: The guard is on the CALLBACK, not on _fetchAndRender itself. This way, the initial _fetchAndRender() call inside switchTo still works (it's called directly, not through the callback). Only coordinator-triggered re-renders during the switch are dropped.

    Wait — there's a subtlety. The coordinator subscription is created DURING switchTo (step 5), and the `_isSwitching` flag is true at that point. The subscription callback fires when `coordinator.scheduleUpdate()` is called. During switchTo, after subscribing (step 5), the code calls `_fetchAndRender()` directly (step 6). The `_isSwitching` flag is still true, but that direct call bypasses the guard since it's not going through the callback.

    The scenario we're guarding against: Between steps 1-6, an EXTERNAL coordinator notification fires (e.g., from pafv.setViewType at step 3 which triggers the old subscription before teardown, or from StateManager.setActiveDataset resetting providers). But actually, the old subscription IS torn down in step 1 (_teardownCurrentView unsubs), and the new subscription is created in step 5. Between steps 1 and 5, there IS no active subscription, so no notifications can fire.

    The real risk is: step 3 (`pafv.setViewType`) triggers a synchronous coordinator notification BEFORE step 1's unsubscribe takes effect (impossible — step 1 runs first). Or: the new subscription (step 5) receives a QUEUED notification from before the switch. If coordinator uses microtask/setTimeout scheduling, a notification queued before teardown could fire after the new subscription is set up but before switchTo completes.

    The guard handles this: any coordinator callback that fires while `_isSwitching` is true (i.e., during switchTo) is silently dropped. The initial `_fetchAndRender()` at step 6 still runs because it's called directly.
