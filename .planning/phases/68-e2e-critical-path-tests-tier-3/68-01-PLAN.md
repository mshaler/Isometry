  <files>e2e/view-switch.spec.ts</files>
  <action>
Create `e2e/view-switch.spec.ts` following the established Tier 2 spec pattern:

1. Import `{ test, expect }` from `'./fixtures'` (shared baseline fixture)
2. Create a `test.describe('Flow 1: View switching preserves card count')` block
3. Write a test that:
   a. Asserts `baselineCardCount > 0` (precondition)
   b. Loops through all 9 view types: `['list', 'grid', 'kanban', 'calendar', 'timeline', 'gallery', 'network', 'tree', 'supergrid']`
   c. For each view:
      - Switch via `page.evaluate(async (vt) => { const { viewManager, viewFactory } = (window as any).__isometry; await viewManager.switchTo(vt, () => viewFactory[vt]()); }, view)`
      - Wait for the view tab to show as active: `page.waitForFunction((viewName) => { const tab = document.querySelector('.view-tab--active'); return tab?.textContent?.toLowerCase().includes(viewName.toLowerCase()); }, view, { timeout: 15_000 })`
      - Query card count from database via `page.evaluate(async () => { const { bridge } = (window as any).__isometry; const r = await bridge.send('db:query', { sql: 'SELECT COUNT(*) as cnt FROM cards WHERE deleted_at IS NULL', params: [] }); return r.rows[0]?.cnt ?? 0; })`
      - Assert count equals `baselineCardCount`
   d. After the loop, verify no loading spinner is visible: `expect(page.locator('.loading-spinner:visible, .view-loading:visible')).toHaveCount(0)`

**Important patterns:**
- Use `page.waitForFunction()` with polling (not bare `waitForTimeout`) for tab assertion
- For Network view, add extra wait (2000ms) after tab assertion because force simulation needs time to stabilize — but the card count check via SQL is independent of render completion
- For SuperGrid, the card count is still the database ground truth (not the DOM cell count), so the same SQL query works

**DO NOT:**
- Use `switchToView()` helper from `e2e/helpers/isometry.ts` — it adds 800ms wait but doesn't verify tab activation. Use inline `page.evaluate()` + `waitForFunction()` for the explicit assertion chain per the existing Tier 2 spec pattern.
- Hard-code expected card counts — use `baselineCardCount` from fixture.

<task type="auto">
  <name>Task 2: E2E spec for Flow 11 — Compound Filter Conjunction Correctness</name>
  <files>e2e/compound-filter.spec.ts</files>
  <action>
Create `e2e/compound-filter.spec.ts` following the established Tier 2 spec pattern:

1. Import `{ test, expect }` from `'./fixtures'`
2. Create a `test.describe('Flow 11: Compound filter conjunction correctness')` block
3. Write a test that:
   a. Assert `baselineCardCount > 0`
   b. Switch to SuperGrid: `page.evaluate(async () => { const { viewManager, viewFactory } = (window as any).__isometry; await viewManager.switchTo('supergrid', () => viewFactory['supergrid']()); })`
   c. Wait for grid to render (data cells or headers visible)
   d. Capture initial card count from database as `count0`

   **Step 1: Category filter (folder = Film)**
   e. Apply: `page.evaluate(() => { const { filter, coordinator } = (window as any).__isometry; filter.setAxisFilter('folder', ['Film']); coordinator.scheduleUpdate(); })`
   f. Wait 500ms for re-render
   g. Capture `count1` via SQL COUNT query
   h. Assert `count1 < count0` (filter narrowed)

   **Step 2: Text search (FTS5 search for a term)**
   i. Apply: `page.evaluate(() => { const { filter, coordinator } = (window as any).__isometry; filter.setSearchQuery('Oscar'); coordinator.scheduleUpdate(); })`
   j. Wait 500ms
   k. Capture `count2`
   l. Assert `count2 <= count1` (conjunction further narrows — use lessThanOrEqual because search might match all Film cards)

   **Step 3: Range filter (priority range)**
   m. Apply: `page.evaluate(() => { const { filter, coordinator } = (window as any).__isometry; filter.setRangeFilter('priority', 3, 7); coordinator.scheduleUpdate(); })`
   n. Wait 500ms
   o. Capture `count3`
   p. Assert `count3 <= count2` (triple conjunction)

   **Step 4: Verify filter compile() state**
   q. Read compiled filter: `filter.compile()` — assert `where` contains `folder IN (?)` AND FTS/LIKE clause AND priority range clause
   r. Assert `params` includes `'Film'`

   **Step 5: Remove one filter → results broaden**
   s. Clear axis filter: `page.evaluate(() => { const { filter, coordinator } = (window as any).__isometry; filter.setAxisFilter('folder', []); coordinator.scheduleUpdate(); })`
   t. Wait 500ms
   u. Capture `count4`
   v. Assert `count4 >= count3` (removing one filter broadens)

   **Step 6: Clear all → full restoration**
   w. Clear everything: `page.evaluate(() => { const { filter, coordinator } = (window as any).__isometry; filter.clearFilters(); coordinator.scheduleUpdate(); })`
   x. Wait 500ms
   y. Capture `countFinal`
   z. Assert `countFinal === count0` (full restoration)

**Important:** Use `bridge.send('db:query', { sql: 'SELECT COUNT(*) ...' })` for all card counts — this is the ground truth, independent of what the grid renders. The search term `'Oscar'` works well because the Meryl Streep dataset has Oscar-related content in multiple cards.

**DO NOT:**
- Use DOM cell counts for data assertions (SuperGrid has its own cell rendering that doesn't map 1:1 to card count)
- Assume exact counts — only assert relative ordering (less than, less than or equal, greater or equal)
