**Section 0 (NEW — per D-03): "When to use this template"**

Add a section before section 1 with the heading `## 0. When to use this template`. Content must include:
- One paragraph explaining this template is for **operator surfaces** — components that produce typed configuration consumed by other views, rather than projecting data onto spatial planes
- A two-row comparison table with columns `| Template | Use when | Example |`:
  - Row 1: `geometry-contract-template.md` / "The component owns a coordinate system, binds axes to PAFV planes, and projects data onto spatial coordinates" / "Time Explorer — projects temporal facets onto an x-plane via `scaleTime`, produces membership predicates from brush selections"
  - Row 2: `operator-contract-template.md` / "The component produces typed configuration or transformation rules consumed by downstream views, with no spatial projection of its own" / "Chip Well — hosts draggable chip tokens organized into typed categories, publishes ordered DSL fragments to a compilation pipeline"
- A cross-link sentence: "For components with spatial projection semantics, use the sibling [geometry contract template](geometry-contract-template.md) instead."

**Section 1: Intent** — Keep section number and heading. Replace the geometry-specific italicized guidance prompt with:
*"One or two paragraphs of plain prose. What does this operator surface do? What typed configuration does it produce? What question is the user answering when they interact with it? What makes it architecturally distinct from a data view — why does it not project data onto spatial planes?"*

Keep the SuperTime example placeholder but replace it with an operator-surface example placeholder:
"[Component name] is the [purpose] surface. It produces [output type] consumed by [downstream consumers]. It is not a data view — it does not project data onto spatial planes or answer spatial questions. Its output is typed configuration that [what downstream does with it]."

**Section 2 (REPLACES geometry sections 2 AND 3 — per D-01): "Operator Surface"**

Replace both `## 2. Coordinate system` and `## 3. PAFV binding` with a single `## 2. Operator surface` section. This section has three subsections:

- **### Input contract** — *"What typed inputs does this operator accept? What is the shape of each input (type signature, cardinality, ordering)? Reference the schema or type definitions. What must be true about the input for the operator to function?"*
  Include a TypeScript interface placeholder block.

- **### Output contract** — *"What typed output does this operator produce? What downstream consumers read this output? Is the output read-only from the consumer's perspective or bidirectional?"*
  Include a TypeScript interface placeholder block.

- **### Why this is not geometric** — *"One or two sentences explaining why this component does not bind to PAFV planes or own a coordinate system. This is the architectural boundary statement — it signals to future readers that the absence of spatial semantics is deliberate, not an oversight. Reference FE-RG-13 if applicable."*

**Sections 3-11 (renumbered from geometry template sections 4-12 — per D-02):**

Carry over all 10 remaining sections from the geometry template but renumber them (geometry section 4 becomes section 3, etc.) and rewrite each section's italicized guidance prompt to be operator-aware. Specific rewrites:

- **Section 3 (was 4): Data binding** — Change "What shape of data does this view accept?" to "What runtime data does this operator surface bind to beyond its typed input contract? (If §2 fully describes the data flow, state 'Covered in §2 — no additional data binding.' Otherwise, describe supplementary data sources.)"
  Keep the cardinality table but change regimes from spatial rendering to operator processing (e.g., "Empty: 0 inputs — empty state", "Sparse: 1-5 inputs — single pass", "Typical: 5-20 inputs — standard processing", "Dense: 20+ inputs — consider batching or pagination").
  Keep the aggregate handling subsection but change the prompt to "Does this operator surface display or compute aggregates? If not, state 'N/A — aggregates are produced downstream by [consumer].'"

- **Section 4 (was 5): Invariants** — Change examples from spatial invariants to operator invariants:
  - "Every output item corresponds to exactly one user-configured input. No phantom outputs."
  - "Input order is user-controlled and semantically meaningful. The system never silently reorders."
  - "The operator is idempotent: applying the same configuration twice produces the same output."

- **Section 5 (was 6): Degenerate and edge cases** — Keep the table format. Replace spatial edge cases with operator edge cases:
  - "Empty input" / "Type-incompatible input" / "Duplicate inputs" / "Input referencing deleted or missing upstream artifact" / "Circular dependency among inputs" / "Maximum input cardinality exceeded"

- **Section 6 (was 7): States** — Keep reflow rules, lifecycle states, empty/loading/error state subsections. Change the lifecycle state machine transitions from spatial (Rendered/Rebinding) to operator (Configured/Reconfiguring). Replace the mermaid diagram with:
  ```
  [*] --> Loading
  Loading --> Empty: inputs.length === 0
  Loading --> Configured: inputs.length > 0
  Configured --> Reconfiguring: input added/removed/reordered
  Reconfiguring --> Configured: output recomputed
  Configured --> Error: validation fails
  Empty --> Configured: input arrives
  ```

- **Section 7 (was 8): Interaction contracts** — Keep gesture table, keyboard contract, accessibility contract subsections. Change the example gesture from "drag facet chip to plane" to "drag item into operator surface". Add a note about WKWebView pointer-event constraint where relevant.

- **Section 8 (was 9): Composition** — Keep upstream/downstream/coordination subsections. Change guidance from spatial composition ("What selections, filters, or state does this view read from elsewhere?") to operator composition ("What upstream state does this operator consume? What downstream consumers read its output? What coordination guarantees exist with sibling operators?").

- **Section 9 (was 10): Out of scope** — Keep the list format. Replace spatial examples with operator examples:
  - "Visual styling (colors, typography, spacing) — see design tokens"
  - "DSL grammar or compilation — see compilation pipeline spec"
  - "Spatial projection of operator output — see consuming view's geometry contract"
  - "Per-consumer-specific rules — see each consumer's spec"

- **Section 10 (was 11): Open questions** — Keep format unchanged.

- **Section 11 (was 12): Reference** — Keep visual calibration, related contracts, changelog subsections. Change "Related contracts" guidance to mention both geometry contracts (for contrast) and other operator contracts (for consistency).

**Header block:** Match the geometry template's header style but adapted:
```
# Operator Contract: `[ComponentName]`

*Status: Draft | Review | Approved*
*Owner: [name]*
*Last updated: <date>*
*Related: [links to upstream specs, consuming view geometry contracts]*

> **What this document is.** A specification of the *operator function* that this surface performs — what typed configuration it produces and what downstream consumers it serves. If this document is doing its job, a redesign of visual chrome should not invalidate it; only a change to operator semantics, input/output contracts, or composition rules should.
>
> **What this document is not.** A geometry contract. A component spec. A design mockup. A style guide. Spatial projection semantics live in the consuming view's geometry contract, not here.
```

The tone, depth, and formatting conventions must match the geometry template exactly — same use of italicized guidance prompts, same placeholder patterns, same table formats.
