    Note: this uses the coordinator's existing subscription mechanism. LatchExplorers.update() will re-fetch distinct values if its dirty flag is set. The coordinator fires on every scheduleUpdate() -- which includes after imports. LatchExplorers should internally track whether values need refresh (dirty flag set by coordinator subscription, actual fetch deferred to update()).

    Actually, looking at the design more carefully: the coordinator subscription in main.ts already fires on every state change (filter, pafv, density changes). LatchExplorers should NOT re-fetch distinct values on every filter change -- only after imports. Pass the coordinator reference to LatchExplorers via config and let LatchExplorers subscribe internally to coordinator with its own dirty-flag logic. The config already includes `coordinator`. LatchExplorers will subscribe to coordinator in mount() and set a `_valuesDirty` flag. The update() method checks this flag and re-fetches if true.

    So the main.ts wiring is simpler -- just mount LatchExplorers with the config that includes coordinator, and LatchExplorers handles its own subscription internally.
