**Step 1: Modify ViewTabBar to accept optional mountTarget.**

Add an optional `mountTarget` property to ViewTabBarConfig:

```typescript
interface ViewTabBarConfig {
  container: HTMLElement;
  onSwitch: (viewType: ViewType) => void;
  mountTarget?: HTMLElement;  // If provided, append to this instead of insertBefore
}
```

In the constructor, change the mounting logic:
```typescript
// OLD: config.container.parentElement?.insertBefore(this._el, config.container);
// NEW:
if (config.mountTarget) {
  config.mountTarget.appendChild(this._el);
} else {
  // Backward compat: keep insertBefore for any code not yet using mountTarget
  config.container.parentElement?.insertBefore(this._el, config.container);
}
```

This is a backward-compatible change — existing code without mountTarget still works.

**Step 2: Re-wire main.ts to create WorkbenchShell and re-root ViewManager.**

In the main() function, after getting `container` (the #app element), create WorkbenchShell:

```typescript
// After: const container = document.getElementById('app');

// Create WorkbenchShell — takes over #app, creates .workbench-shell layout
const shell = new WorkbenchShell(container, {
  commandBarConfig: {
    onOpenPalette: () => {
      if (commandPalette.isVisible()) {
        commandPalette.close();
      } else {
        if (helpOverlay.isVisible()) helpOverlay.hide();
        commandPalette.open();
      }
    },
    onCycleTheme: () => {
      const modes: ThemeMode[] = ['dark', 'light', 'system'];
      const current = modes.indexOf(theme.theme);
      const next = modes[(current + 1) % modes.length]!;
      theme.setTheme(next);
    },
    onCycleDensity: () => {
      const modes: DensityMode[] = ['compact', 'comfortable', 'spacious'];
      const current = modes.indexOf(density.mode);
      const next = modes[(current + 1) % modes.length]!;
      density.setMode(next);
    },
    onToggleHelp: () => {
      helpOverlay.toggle();
    },
    getThemeLabel: () => theme.theme.charAt(0).toUpperCase() + theme.theme.slice(1),
    getDensityLabel: () => density.mode.charAt(0).toUpperCase() + density.mode.slice(1),
  },
});
```

**IMPORTANT:** The shell constructor uses commandPalette, helpOverlay, theme, density — but these are defined later in main.ts. To solve this, the WorkbenchShell must be created AFTER providers and overlays are created, OR the callbacks must be wired after creation. The cleanest approach: move WorkbenchShell creation to AFTER line ~140 (after helpOverlay, commandPalette, theme, density are all created). The ViewManager creation will need to be reordered too.

**Key change: AuditOverlay — mount toggle button to document.body, but toggle .audit-mode on container (#app):**
The AuditOverlay.mount(container) currently both appends the button AND toggles .audit-mode on `container`. After the re-root, the button should go to document.body for z-index layering, but .audit-mode must stay on the #app element (or .workbench-shell). Since AuditOverlay toggles .audit-mode on the container it received, pass `container` (the original #app) to auditOverlay.mount() — this keeps audit CSS working. The button renders inside #app which is fine since .workbench-shell is a child of #app.

Actually, per RESEARCH.md recommendation: "AuditOverlay toggle button mounts to document.body but toggles .audit-mode on the original #app element." However, modifying AuditOverlay to accept two containers is scope creep. Simpler: keep `auditOverlay.mount(container)` — the button renders in #app (outside .workbench-shell since it appends to container directly), and .audit-mode toggles on container. This works because #app contains .workbench-shell, and the fixed-position button uses `position: fixed` so its visual position is unaffected by parent layout.
