**Section 1: Live Preview Behavior (UXIN-01)**

Must contain these subsections:
- §1.1 Trigger: Every chip drop, reorder, or remove fires re-compilation immediately (no debounce). Trigger events: `chip:drop`, `chip:reorder`, `chip:remove`. Per D-01 from CONTEXT.md.
- §1.2 Pipeline path: Chip arrangement → `ChipWellOutputContract` (from 06-chip-well-geometry.md §9) → `FormulasProvider.compile()` → `(sql_text, bind_values)` tuple → Worker bridge `supergrid:query` message → sql.js execution → result set returned to main thread.
- §1.3 Result animation: D3 keyed data-join transitions. Rows enter with opacity 0→1 + translateY(8px→0). Rows exit with opacity 1→0. Rows update reposition with 200ms ease-out. Key function uses row primary key. Per D-02, consistent with existing SuperGrid/TreeView/NetworkView transitions.
- §1.4 Loading state: If query exceeds 200ms threshold, show inline skeleton overlay (CSS shimmer on `.chip-well-preview--loading`). Current results stay visible but dimmed (opacity 0.5). Overlay removed on result arrival. Per D-03.
- §1.5 Sequencing diagram: Text-based sequence showing: User drops chip → compile() → Worker message → SQL execution → D3 transition → preview visible. Include branching path for >200ms (shimmer overlay appears).

**Section 2: Reversibility Model (UXIN-02)**

Must contain:
- §2.1 Scope: Per-well undo stack. Each chip well (`Calculations`, `Filters`, `Sorts` in Formulas; `Conditional Encoding` in Marks; `Anomaly Rules`, `Validation Rules` in Audits) maintains its own stack. Per D-04.
- §2.2 Stack entry shape: `ArrangementSnapshot = { wellId: string, chips: ChipState[], timestamp: number }`. Each snapshot is the complete chip list for that well at that point.
- §2.3 Stack depth: Unlimited in-memory. Not persisted across sessions. Cleared on navigation away from the explorer. Per D-05.
- §2.4 Undo trigger: Cmd+Z (macOS) / Ctrl+Z. Only operates on the focused well. If no well is focused, no-op.
- §2.5 Redo trigger: Cmd+Shift+Z / Ctrl+Shift+Z. Standard redo semantics (redo stack cleared on new action).
- §2.6 Undo fires re-preview: Reverting an arrangement snapshot triggers the same compile → D3 transition pipeline as a new chip drop. Per D-06.
- §2.7 Explicit non-support: Formula Card save (`promoteToCard`) is NOT undoable. Saving creates a permanent version row. This is stated as a design boundary, not a missing feature. Per D-07.
- §2.8 Boundary table: A table with columns [Action | Undoable? | Mechanism | Notes] listing: chip drop (yes, arrangement stack), chip reorder (yes, arrangement stack), chip remove (yes, arrangement stack), Formula Card save (NO, permanent version), chip rename/edit DSL (yes, arrangement stack).

**Section 3: Error State Presentation (UXIN-03)**

Must contain:
- §3.1 Error taxonomy: Three error types from Phase 184 compilation pipeline:
  1. Type mismatch — chip's type signature incompatible with well or dependent chip
  2. Dependency cycle — `CycleError` from topological sort (participants field carries chip IDs)
  3. Compilation error — DSL syntax error, invalid column reference, or bind-value failure
- §3.2 Wireframe: Type mismatch — Single chip gets `data-error="type-mismatch"` attribute, red 2px outline (`--color-error`), error icon (⚠) appended to chip label. Tooltip on hover: "Type mismatch: expected {expected}, got {actual}". Tooltip includes "Undo" action link.
- §3.3 Wireframe: Dependency cycle — ALL chips in `CycleError.participants` array get `data-error="cycle"` attribute, red 2px outline, circular-arrow icon. Tooltip: "Dependency cycle: {chip-A} ↔ {chip-B} [↔ ...]". Each participating chip shows the same tooltip. "Undo" action link included.
- §3.4 Wireframe: Compilation error — Offending chip gets `data-error="compile"` attribute, red 2px outline, X icon. Tooltip: "{error.message}" (from compiler). "Undo" action link included.
- §3.5 Partial results behavior: Valid chips compile and execute normally. Erroring chips are excluded from the query. Preview shows partial results with an indicator bar: "Showing partial results — {N} chip(s) excluded due to errors". Per D-10.
- §3.6 Error clearing: Errors clear automatically when the causing condition is resolved (chip removed, reordered to break cycle, type signature corrected by upstream change).
- §3.7 Multiple simultaneous errors: Each chip shows its own error independently. Multiple chips can be in error state simultaneously. No global error banner — errors are always spatially co-located with the causing chip. Per D-08.
- §3.8 Recovery actions: Every error tooltip includes "Undo" link that triggers `well.undo()`. Per D-09. Additional recovery for cycle errors: tooltip also shows which chips to separate.

**Section 4: Save-as-Formula Promotion UI Flow (UXIN-04)**

Must contain:
- §4.1 Trigger: "Save as Formula" button in chip-well toolbar. Always visible when at least one chip is in the well. Button uses `<button>` element with `data-action="promote"`. Per D-11.
- §4.2 Pre-validation: On click, call `validatePromotion(currentArrangement)` (from 03-formula-card-schema.md §4). If validation fails (e.g., empty arrangement, all chips in error state), show inline error on the button ("Cannot save: {reason}") and do NOT open dialog. Per D-12.
- §4.3 Dialog specification: `<dialog>` element (per D-14, D-19). Modal. Contents:
  - Title: "Save as Formula"
  - Field 1: Name (text input, required, placeholder "e.g., Revenue Growth Filter", max 100 chars, auto-focused)
  - Field 2: Description (textarea, optional, placeholder "What does this formula do?", max 500 chars)
  - Hidden defaults: `visibility = 'active'`, `scope = 'dataset'`
  - Buttons: "Cancel" (secondary) | "Save" (primary, disabled until Name has content)
- §4.4 Confirm flow: On "Save" click → call `promoteToCard({ title, description, arrangement })` → on success: close dialog, show toast "Formula saved" (using existing toast pattern), chips remain in wells as-is (per D-13). Arrangement is now both live preview AND a saved card.
- §4.5 Cancel flow: On "Cancel" or Escape key or backdrop click → close dialog, no state change, chips remain as-is.
- §4.6 Post-save editing: User can continue editing chips after save. Next "Save as Formula" creates a new version row (per versioning strategy in 03-formula-card-schema.md §3). The arrangement is not "locked" after save.
- §4.7 Error during save: If `promoteToCard()` throws (e.g., SQLite constraint violation), show error in dialog footer: "Save failed: {error.message}". Dialog stays open. User can retry or cancel.

**Section 5: Explorer Placement in Navigation (UXIN-05)**

  1. **Formulas** — Calculations, Filters, Sorts chip wells (data-layer operations)
  2. **Marks** — Conditional Encoding chip well (view-layer CSS class assignments)
  3. **Audits** — Anomaly Rules, Validation Rules chip wells (semantic flags)

- §5.4 Orthogonality statement: Formulas Explorer is NOT the "A" (Analyze) in LATCH. It is an orthogonal operator surface that produces configuration consumed by views. LATCH governs data taxonomy; Formulas governs data transformation. Per FE-RG-11 and D-16.
- §5.5 Coexistence: Formulas/Marks/Audits explorers coexist with other Analyze-section explorers (e.g., CalcExplorer, HistogramExplorer). They do not replace or subsume existing explorers. They add a new chip-well-based interaction surface alongside the existing panel-based explorers.

**Section 6: WKWebView Constraint Compliance (UXIN-06)**

Must contain:
- §6.1 Constraint checklist table with columns [Constraint | Rationale | Implementation Rule | Violation Detector]:
  1. Pointer events for chip drag | HTML5 DnD API unreliable in WKWebView (drops silently fail) | Use `pointerdown`/`pointermove`/`pointerup` event sequence. Manual hit-testing via `elementFromPoint()`. Per D-17. | grep for `dragstart|dragend|draggable` in chip-well code = violation
  2. No `:has()` behavioral selectors | Safari versions in WKWebView have inconsistent `:has()` support for JS-triggered state | Use `data-*` attribute patterns instead (e.g., `data-error`, `data-drag-state`). Per D-18. | grep for `:has(` in CSS that targets chip-well elements = violation
  3. `<dialog>` for all prompts | `alert()`/`confirm()`/`prompt()` are blocked or unreliable in WKWebView | All modal interactions use `<dialog>` element with `showModal()`/`close()`. Per D-14, D-19. | grep for `alert\(|confirm\(|prompt\(` in formulas code = violation
- §6.2 Pointer event protocol: Detailed sequence for chip drag within WKWebView:
  - `pointerdown` on chip → capture pointer (`setPointerCapture`), record start position, set `data-drag-state="dragging"` on chip
  - `pointermove` → update chip ghost position via `transform: translate()`, hit-test drop zones via `elementFromPoint()`
  - `pointerup` → release capture, determine drop target, fire `chip:drop` or `chip:reorder` event, clear `data-drag-state`
  - Touch-action: `touch-action: none` on chip elements to prevent scroll interference
- §6.3 Cross-reference: Spatial drag affordances (hit regions, drop zones, visual feedback) defined in `06-chip-well-geometry.md` §5 and §7. This section specifies only the event-level protocol; geometry contract owns spatial rules.
