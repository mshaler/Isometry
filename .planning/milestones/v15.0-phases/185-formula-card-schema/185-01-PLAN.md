**Section 2 — Type-Signature Validation Algorithm (CARD-02):**

Write the complete type-signature validation algorithm with pseudocode. Cover:

- **Existing facet types mapped to signature types:** text, number (integer + real), date (all temporal columns), boolean (integer 0/1), select (text with CHECK constraint or tag membership), multi_select (JSON array text).
- **Type compatibility matrix:** A table showing which input types are compatible with which operations (e.g., number inputs accept arithmetic operators; text inputs accept string functions; date inputs accept temporal functions).
- **Validation timing:** runs at chip-drop time (before execution), not at save time. When a chip is dragged into a well, the type signature of the chip is validated against the well's expected input types.
- **Algorithm pseudocode:**

```
function validateTypeSignature(
  chipSignature: TypeSignature,
  wellCategory: WellCategory
): ValidationResult {

  // Step 1: Check output type compatibility with well category
  //   - Filters wells require output.type == 'boolean'
  //   - Sorts wells require any scalar output type (text, number, date)
  //   - Calculations wells accept any output type
  //   - Marks/Conditional encoding wells require output.type == 'boolean'
  //   - Audits wells require output.type == 'boolean'

  // Step 2: Check each input column exists in SchemaProvider
  //   - For each input in chipSignature.inputs:
  //     - Validate column name against SchemaProvider allowlist (COMP-04, FE-RG-17)
  //     - Validate column type matches declared input type

  // Step 3: Return ValidationResult
  //   - { valid: true } or { valid: false, errors: ValidationError[] }
}
```

- **Extensibility mechanism:** Explain how to add new types (array, JSON, geo-shape) without modifying the core algorithm:
  1. Add the new type string to the TypeSignature type union (`'text' | 'number' | 'date' | 'boolean' | 'array' | 'json' | 'geo_shape'`)
  2. Add a row to the type compatibility matrix defining which well categories accept the new type as output
  3. Add input-type validation logic for the new type (e.g., geo_shape inputs require latitude/longitude column pairs)
  4. The core algorithm structure (3-step: output check, input check, return) is unchanged

- **Worked examples:** At least 4 examples covering:
  1. A numeric Calculation (`revenue - cost AS profit`) — inputs: [revenue: number, cost: number], output: number → valid in Calculations well
  2. A boolean Filter (`priority > 3`) — inputs: [priority: number], output: boolean → valid in Filters, Marks, Audits wells
  3. A text Sort (`name ASC`) — inputs: [name: text], output: text → valid in Sorts well
  4. A type mismatch — a text-output expression dropped into a Filters well → invalid (Filters require boolean output)

**Section 3 — Versioning Strategy (CARD-03):**

Document the immutable-version-row strategy:

- Every save creates a new row with a new `id` (UUID), the same `canonical_id`, and `version` incremented by 1.
- `canonical_id` is the stable identity used everywhere: in `dependencies` arrays, in cross-version references, in the chip well's reference to its source card.
- `id` is the version-specific identity — unique per version row.
- **Version query pattern:** To get the latest version of a card: `SELECT * FROM formula_cards WHERE canonical_id = ? AND deleted_at IS NULL ORDER BY version DESC LIMIT 1`.
- **All versions retained** at v1. No coalescing or pruning policy (per STATE.md locked decision).
- **Version history query:** `SELECT id, version, title, created_at FROM formula_cards WHERE canonical_id = ? ORDER BY version ASC` — returns the complete version timeline for a card.
- **Rollback:** To revert to a previous version, create a new version row (version N+1) that copies the content from the target historical version. The historical row is never mutated.

**Section 5 — Sync Conflict Resolution (CARD-05):**

Document the keep-both (version fork) strategy per D-03 and D-04 for all three scenarios. Emphasize this is a deliberate departure from the last-writer-wins strategy used for cards/connections (src/native/NativeBridge.ts SyncMerger).

**Scenario 1 — Concurrent edit:**
- Device A saves version N+1 of canonical_id X. Device B saves a different version N+1 of the same canonical_id X.
- Resolution: Both version rows are retained. Both get version N+1 but with different `id` values. A `conflict_marker` flag (or provenance metadata) indicates the fork.
- User resolution: The Formulas Explorer shows conflicted versions and the user picks one (or merges manually). The chosen version becomes version N+2.

**Scenario 2 — Delete-while-editing:**
- Device A soft-deletes canonical_id X (sets deleted_at). Device B saves a new version of canonical_id X.
- Resolution: The edit survives as a "resurrection" version row — the new version has `deleted_at = NULL`. The user sees both states and confirms whether to keep the edit or honor the delete.

**Scenario 3 — Type-signature change:**
- Device A changes the type_signature of canonical_id X (e.g., adds an input column). Device B saves a version with the old type_signature.
- Resolution: Both type-signature variants are retained as separate version rows. The user resolves which signature is canonical. Downstream dependencies referencing canonical_id X are validated against the resolved signature.

**Uniform strategy rationale (D-04):** All three scenarios use the same keep-both mechanism. No scenario-specific branching in the sync logic. The sync engine treats formula_cards identically regardless of which fields conflict — it always forks. User resolution is always the same: pick one version, which becomes the new latest.
