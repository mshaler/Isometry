# Phase 165 — UI Design Contract: Canvas Stubs + Registry

## Phase Character

Phase 165 is a **pure TypeScript implementation phase**. It produces zero new user-visible UI surfaces. The three stub classes (ExplorerCanvasStub, ViewCanvasStub, EditorCanvasStub) and the canvas registry module are internal scaffolding consumed by Phase 166 integration tests and replaced in v13.1+.

The visual contract below governs the **data-attribute surface** (the DOM contract that tests assert against) and the **debug text content** that stubs render as a human-readable aid during integration testing. No end-user-facing copy or interactive states exist in this phase.

## DOM Contract (Data Attributes)

The binding contract for Phase 165 is expressed as data attributes, not visual styles. These are what tests assert and what Phase 166 integration tests verify.

### ExplorerCanvasStub (CANV-01)

| Attribute | Value | Notes |
|-----------|-------|-------|
| `data-canvas-type` | `"Explorer"` | Set on stub root element by mount() |
| `data-render-count` | `"1"`, `"2"`, ... | String; increments each mount() call, persists across destroy/mount cycles |
| text content | `[Explorer: {canvasId}]` | Debug aid; not a user-visible label |

### ViewCanvasStub (CANV-02)

| Attribute | Value | Notes |
|-----------|-------|-------|
| `data-canvas-type` | `"View"` | Set on stub root element by mount() |
| `data-render-count` | `"1"`, `"2"`, ... | String; same increment contract as Explorer |
| `data-sidecar` (child element) | `""` (present) | Present when binding = `'Bound'`; absent when binding = `'Unbound'` |
| text content | `[View: {canvasId}]` | Debug aid |

The sidecar element is a child `div` of the stub's root element. It is NOT a sibling in the canvas slot. ViewCanvasStub manages sidecar DOM internally (D-04).

### EditorCanvasStub (CANV-03)

| Attribute | Value | Notes |
|-----------|-------|-------|
| `data-canvas-type` | `"Editor"` | Set on stub root element by mount() |
| `data-render-count` | `"1"`, `"2"`, ... | Same increment contract |
| No sidecar | — | Editor is Unbound-only; no sidecar affordance |
| text content | `[Editor: {canvasId}]` | Debug aid |

### data-render-count semantics

`data-render-count` on the stub root element is **distinct** from the slot-level `data-render-count` managed by SuperWidget (which tracks canvas-slot re-renders). Stub render count:

- Starts at `0` before first mount (attribute not yet set)
- After first `mount()`: `"1"`
- After `destroy()` + second `mount()`: `"2"` (does NOT reset on destroy)
- All values are strings (DOMStringMap); test assertions must use `toBe('1')`, not `toBe(1)`

### registerAllStubs() canonical wiring (D-03)

| canvasId | canvasType | defaultExplorerId | Notes |
|----------|------------|-------------------|-------|
| `'explorer-1'` | `'Explorer'` | — | ExplorerCanvasStub |
| `'view-1'` | `'View'` | `'explorer-1'` | ViewCanvasStub — satisfies CANV-05 |
| `'editor-1'` | `'Editor'` | — | EditorCanvasStub |

`registerAllStubs()` lives in `registry.ts` (not a separate stubs barrel) — reduces files, registry already owns the Map. Source: RESEARCH.md open question 2 resolution.

### Registry isolation

`_registry` is a module-level Map. Tests must call `clearRegistry()` (exported from registry.ts, test-only) in `beforeEach` to prevent cross-test state leakage. Source: RESEARCH.md open question 1 resolution.
