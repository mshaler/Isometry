  The key pattern: two sequential tests where Test A mutates shared state and Test B (running on a fresh harness) asserts defaults. Since each `makePluginHarness()` call runs `registerCatalog()` which creates NEW shared state objects via the create*State() factories, isolation is by construction. The test proves this.

  Test: "Test A — mutate shared state via plugin interactions"
  - Create harness, enable zoom plugins (`superzoom.slider`, `superzoom.scale`), select plugins (`superselect.click`), search plugins (`supersearch.input`), density plugins (`superdensity.mode-switch`), stack plugins (`superstack.spanning`, `superstack.collapse`), audit plugins (`superaudit.overlay`)
  - Run pipeline to exercise all hooks
  - This test exists solely to prove the NEXT test sees clean state

  Test: "Test B — fresh harness has clean default state (no leakage from Test A)"
  - Create a NEW harness via `makePluginHarness()`
  - Enable each category's root plugin one at a time, access the plugin instance via bracket notation `(harness.registry as any)._plugins.get(pluginId)?.instance`
  - For each shared state category, verify default values:
    - ZoomState: enable `superzoom.scale`, access its hook — after runPipeline, layout.zoom should equal 1 (default, not mutated)
    - SelectionState: enable `superselect.click` — no cells should be in selection
    - SearchState: enable `supersearch.input` — search query should be empty/default
    - DensityState: enable `superdensity.mode-switch` — density level should be default
    - SuperStackState: enable `superstack.collapse` — collapsedSet should be empty (size 0)
  - The assertions prove no state leaked from Test A's mutations

  Alternative approach (Claude's discretion per CONTEXT.md): If accessing shared state objects directly is impractical due to closure scoping, verify isolation through observable pipeline output instead:
  - Fresh harness with zoom enabled: layout.zoom === 1
  - Fresh harness with scroll: cells length equals input length (no stale windowing)
  - Fresh harness with density: no density-specific meta on cells

  Call `harness.registry.destroyAll()` after each test.
