import Foundation
import CloudKit
@testable import Isometry

**Test strategy:** SyncManager is an actor. `initialize()` creates a real CKSyncEngine which requires CloudKit entitlements -- do NOT call it in tests. Instead, test the file-based persistence layer that `init(appSupportDir:)` loads from and `addPendingChange()` writes to.

**Helper:** Create a `makeTempDir() -> URL` helper that creates a unique temp directory and returns its URL. Each test should use its own temp dir to avoid interference.

**Tests (all async since SyncManager is an actor):**

1. `initWithEmptyDirHasEmptyQueue` -- Create SyncManager with empty temp dir. Verify no crash. Call `consumeReuploadFlag()` -- should return false (needsFullReupload is false because init doesn't call initialize()).

   Wait -- `needsFullReupload` is set to `false` at declaration and only set `true` in `initialize()` when `stateSerialization == nil`. Since we don't call `initialize()`, it stays false. Test this.

2. `consumeReuploadFlagReturnsFalseWithoutInitialize` -- Create SyncManager, call `consumeReuploadFlag()` twice. First call returns false, second also returns false.

3. `addPendingChangePersistsToQueueFile` -- Create SyncManager with temp dir. Call `addPendingChange()` with a save PendingChange. Read the `sync-queue.json` file from the temp dir. Decode as `[PendingChange]`. Verify count == 1 and fields match.

   NOTE: `addPendingChange` also calls `syncEngine?.state.add(...)` which will be nil (no initialize called), so that line is a no-op. The file persistence still happens.

4. `addMultiplePendingChangesAccumulates` -- Add 3 pending changes. Read `sync-queue.json`. Verify count == 3.

5. `pendingChangesRoundTripThroughFileSystem` -- Create SyncManager A with temp dir, add 2 pending changes. Create SyncManager B with SAME temp dir. SyncManager B's init loads from disk. Verify B's queue has 2 items by adding a 3rd change and checking the file has 3 items total.

6. `addPendingChangeDeleteOperation` -- Add a delete PendingChange (operation="delete", fields=nil). Read file. Verify decoded change has operation=="delete" and fields==nil.

7. `queueFileCorruptionReturnsEmptyArray` -- Write garbage bytes to `sync-queue.json` in temp dir. Create SyncManager. Verify no crash (loadQueueFromDisk returns [] on decode failure). Add a pending change -- should work normally.

8. `systemFieldsFileEmptyOnInit` -- Create SyncManager with empty temp dir. Verify `record-metadata.json` does NOT exist (system fields only written after sync events, and we don't call initialize()).

9. `stateFileDoesNotExistOnInit` -- Create SyncManager with empty temp dir. Verify `sync-state.data` does NOT exist.

Each test follows this pattern:
```swift
@Test func testName() async throws {
    let tempDir = makeTempDir()
    defer { try? FileManager.default.removeItem(at: tempDir) }
    let manager = SyncManager(appSupportDir: tempDir)
    // ... test logic (await for actor calls) ...
}
```

IMPORTANT ACTOR NOTES:
- All calls to SyncManager methods must be `await`ed since it's an actor
- `addPendingChange` is `func` (not private), so it's accessible with @testable import
- After `addPendingChange`, the file write is synchronous within the actor, so by the time `await` returns, the file is written

**Test strategy:** NotesAdapter reads from `NoteStore.sqlite` at a hardcoded path. Its `readNotesFromDatabase()` is private and called from `fetchCards()`. The problem is `fetchCards()` uses `Self.noteStorePath` which points to the real NoteStore.

**Solution:** Instead of calling NotesAdapter directly, test the core logic by creating a test-only subclass or by testing the components that NotesAdapter depends on. However, NotesAdapter is a struct, not a class.

**Better approach:** Create a **TestableNotesAdapter** -- a minimal copy of NotesAdapter's `readNotesFromDatabase()` logic that accepts a database path parameter. OR, since the actual SQL queries and schema detection are the critical paths, test them by:

1. Creating a fixture SQLite database in a temp directory with the exact Apple NoteStore schema
2. Opening it with SQLite3 C API and verifying the queries work
3. Testing `ProtobufToMarkdown.extract()` with fixture ZDATA (covered in Plan 01)

**Actually, the best approach:** NotesAdapter's `fetchCards()` calls `readNotesFromDatabase()` which uses `Self.noteStorePath`. Since this is a static computed property, we cannot override it. But we CAN test NotesAdapter's logic by:

- Creating the NotesAdapter and calling `checkPermission()` (will return .denied in test env since no NoteStore.sqlite exists)
- Testing the SQL queries directly against a fixture DB using SQLite3 C API (same approach NotesAdapter uses internally)

**Final strategy:** Create a fixture database, then run the same SQL queries NotesAdapter would run against it, verifying the query logic and schema detection independently. This tests the critical path without needing to inject the path.

**Fixture DB helper:** `makeFixtureNoteStoreDB() -> (URL, OpaquePointer)` that:
- Creates a temp directory
- Opens a new SQLite3 database
- Creates tables matching Apple's NoteStore schema:
