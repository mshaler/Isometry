  <name>Task 1: Same-dimension reorder via HTML5 DnD with insertion position</name>
  <files>src/views/SuperGrid.ts, tests/views/SuperGrid.test.ts</files>
  <behavior>
    - Test: Dropping a col axis onto a different position within the col dimension reorders colAxes (e.g., [A,B,C] with A dropped at index 2 becomes [B,C,A])
    - Test: Dropping a row axis onto a different position within the row dimension reorders rowAxes
    - Test: Dropping an axis at its current position is a no-op (provider setters not called)
    - Test: Same-dimension reorder calls provider.setColAxes (or setRowAxes) with reordered array
    - Test: Same-dimension reorder respects min-1 constraint (reorder within 1-axis dimension is no-op since there's only one axis)
  </behavior>
  <action>
    **Extend the existing drop handler** (from Plan 18-01) to handle same-dimension drops in addition to cross-dimension drops. The dragover handler already gates on `text/x-supergrid-axis`. The drop handler needs to discriminate:

    1. **Cross-dimension** (sourceDimension !== targetDimension): Already implemented by Plan 18-01 — no changes needed.

    2. **Same-dimension** (sourceDimension === targetDimension): New logic. On drop:
       - Read `_dragPayload` (already done by existing handler)
       - Determine the target insertion index from the drop position. Use the drop event's position relative to the header cells to compute which slot the axis should be inserted at. The simplest approach: each header grip/cell has a `data-axis-index` attribute set during rendering. The drop target's `data-axis-index` gives the insertion position.
       - If sourceIndex === targetIndex: no-op (return without calling provider)
       - Otherwise: create new axis array by removing from sourceIndex and inserting at targetIndex
       - Call `this._provider.setColAxes(newAxes)` or `this._provider.setRowAxes(newAxes)` depending on dimension

    **Implementation approach for insertion index detection:**
    - During grip handle rendering (in _createColHeaderCell and row header block), set `el.dataset['axisIndex'] = String(idx)` on each header element.
    - In the drop handler, find the nearest header element from `e.target` (or `e.currentTarget`) and read its `axisIndex` dataset.
    - Alternative: use `e.clientX`/`e.clientY` relative to header strip bounds to compute index from position. The dataset approach is simpler and more reliable.

    **Important:** Same-dimension reorder and cross-dimension transpose use the SAME drag API (HTML5 DnD), the SAME _dragPayload singleton, and the SAME MIME type. The drop handler branches on whether `_dragPayload.sourceDimension` matches the drop zone's dimension. This is the research recommendation: "Use HTML5 DnD for both cross-zone and same-zone reorder. Single API, simpler implementation, consistent behavior."

    **Visual feedback for insertion position:** On dragover within the same dimension, add a visual insertion line (CSS `border-left: 2px solid #3b82f6` or similar) on the header element that would be the drop target. Remove on dragleave/drop.
  </action>

    **Step 1: Add 300ms opacity transition to _fetchAndRender().** Modify the `_fetchAndRender()` method in SuperGrid.ts:

    Before the `_renderCells()` call (after successful bridge response):
    ```typescript

    grid.style.opacity = '0';
    ```

    After the `_renderCells()` call:
    ```typescript

      .transition()
    ```

    This matches the existing crossfadeTransition pattern from transitions.ts (300ms opacity) and the research recommendation: "Set grid opacity to 0 at drop commit time. Set back to 1 after _renderCells() completes."

    **Important nuance:** The opacity transition fires on EVERY _fetchAndRender() call — not just axis transposes. This is correct: any provider change (axis, filter, etc.) that triggers a re-render gets the smooth transition. The first mount call also gets it (subtle fade-in on initial load).

    **Step 2: Handle transition cancelability.** If a second _fetchAndRender() fires while a transition is in-flight (e.g., rapid axis changes), the D3 transition on the same element will automatically cancel the previous one and start fresh. D3 transitions have built-in cancellation on the same element+name — no explicit cancel logic needed.
