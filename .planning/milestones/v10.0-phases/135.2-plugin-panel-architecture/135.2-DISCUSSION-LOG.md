## Panel Plugin Lifecycle

### Q1: How should the panel plugin system relate to the existing SuperGrid PluginRegistry?

| Option | Description | Selected |
|--------|-------------|----------|
| New PanelRegistry (Recommended) | Purpose-built registry with panel-specific hooks (mount/unmount/resize/update). Shares the architectural pattern but with different interfaces. | ✓ |
| Extend PluginRegistry | Add panel hooks to the existing PluginRegistry and PluginTypes. One registry manages both. | |
| Lightweight adapter | No formal registry — just give each explorer a standard interface and let WorkbenchShell manage them directly. | |

**User's choice:** New PanelRegistry (Recommended)
**Notes:** Clean separation from grid plugins preferred.

### Q2: Should the PanelRegistry support panel dependencies?

| Option | Description | Selected |
|--------|-------------|----------|
| All independent (Recommended) | Each explorer panel is standalone. Simplifies toggle logic. | |
| Allow dependencies | Some panels could declare deps. Adds complexity but models real relationships. | ✓ |
| You decide | Claude's discretion. | |

**User's choice:** Allow dependencies
**Notes:** User wants to model real panel relationships.

### Q3: Should panels have an update() hook or subscribe to providers independently?

| Option | Description | Selected |
|--------|-------------|----------|
| Registry update hook (Recommended) | PanelRegistry broadcasts a generic update() to all mounted panels when key providers change. | ✓ |
| Keep independent subscriptions | Each panel continues to subscribe to providers directly. | |
| You decide | Claude's discretion. | |

**User's choice:** Registry update hook (Recommended)
**Notes:** Centralizes the notification pattern.

## Panel Rail vs Sidebar Layout

### Q4: Where does the plugin-based panel system render in the DOM?

| Option | Description | Selected |
|--------|-------------|----------|
| Keep dedicated panel-rail (Recommended) | Panel-rail stays as separate surface. Plugin-ified but same DOM location. | |
| Collapsible panel drawer | Slide-out drawer that toggles open/closed. Hidden by default, opens when panel activated. | ✓ |
| Floating panel windows | Each panel renders as a floating/draggable overlay. | |

**User's choice:** Collapsible panel drawer
**Notes:** User prefers recoverability of screen real estate over always-visible panel rail.

### Q5: When no panels are active, drawer closed state?

| Option | Description | Selected |
|--------|-------------|----------|
| Thin icon strip (Recommended) | ~40px vertical strip with icons for each registered panel. Always visible. | ✓ |
| Fully hidden | Zero width when closed. Panels toggled from SidebarNav or shortcuts only. | |
| You decide | Claude's discretion. | |

**User's choice:** Thin icon strip (Recommended)

### Q6: Should the drawer be resizable or fixed width?

| Option | Description | Selected |
|--------|-------------|----------|
| Fixed width, slides open/closed (Recommended) | Standard ~300px width. CSS transition. No drag handle. | |
| Resizable with drag handle | User drags edge to resize. Width persisted in ui_state. | ✓ |

## Panel Visibility & Toggling

### Q7: How should SidebarNav toggles relate to the new panel icon strip?

| Option | Description | Selected |
|--------|-------------|----------|
| Icon strip is primary (Recommended) | Panel toggles on icon strip only. SidebarNav loses explorer toggles. Clean separation. | ✓ |
