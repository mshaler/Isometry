<objective>
Build SelectionProvider (Tier 3 ephemeral state) and StateCoordinator (cross-provider batching).

Purpose: SelectionProvider manages user selections purely in memory — Cmd+click, Shift+click range, select-all — and is explicitly excluded from any persistence tier. StateCoordinator subscribes to all providers and batches their changes into single notifications for views, preventing redundant queries when multiple providers change in the same frame.

```typescript
private selectedIds = new Set<string>();
private lastSelectedId: string | null = null;

```

Methods:
- `select(id)`: Clear set, add id, set lastSelectedId = id, scheduleNotify()
- `toggle(id)`: If has(id) delete, else add. Update lastSelectedId to id (even on remove, for range). scheduleNotify()
- `range(id: string, allIds: string[])`: Find lastSelectedId index and id index in allIds. Select all between (inclusive). If lastSelectedId not in allIds, fall back to select(id). scheduleNotify()

- `clear()`: Clear set, lastSelectedId = null, scheduleNotify()

- `getSelectedIds()`: return [...selectedIds] (snapshot)
- `getSelectionCount()`: return selectedIds.size
- `subscribe(cb)`: Add to subscribers, return `() => subscribers.delete(cb)`
- `scheduleNotify()`: queueMicrotask with pendingNotify guard

CRITICAL: This class does NOT implement PersistableProvider. It has NO toJSON(), NO setState(), NO resetToDefaults(), NO fromJSON(). If you add any persistence method, it is a bug (violates D-005 and PROV-05).

  <action>
Create `src/providers/StateCoordinator.ts`:

```typescript
interface SubscribableProvider {
  subscribe(cb: () => void): () => void;
}

class StateCoordinator {
  private providerUnsubs = new Map<string, () => void>();
  private subscribers = new Set<() => void>();
  private pendingUpdate: ReturnType<typeof setTimeout> | null = null;

  registerProvider(key: string, provider: SubscribableProvider): void;
  unregisterProvider(key: string): void;
  subscribe(cb: () => void): () => void;
  destroy(): void;
  private scheduleUpdate(): void;
}
```

Key implementation:
- `registerProvider(key, provider)`: Calls provider.subscribe() with a callback that calls scheduleUpdate(). Stores the unsubscribe function.
- `scheduleUpdate()`: If pendingUpdate not null, return (already scheduled). Set pendingUpdate = setTimeout(() => { pendingUpdate = null; notify all subscribers; }, 16).
- `subscribe(cb)`: Adds to subscriber Set, returns removal function.
- `destroy()`: Clears pendingUpdate timer, calls all stored unsubscribe functions, clears maps.

IMPORTANT: Use setTimeout(16), NOT queueMicrotask. Per CONTEXT.md and research Pitfall 6: providers use queueMicrotask for self-notification; StateCoordinator uses setTimeout(16) for cross-provider batching. The setTimeout fires AFTER all microtasks drain, ensuring all providers have settled before coordinator notifies views.

In tests, use `vi.useFakeTimers()` to control setTimeout behavior. Advance by 16ms to trigger coordinator notification.

Update `src/providers/index.ts` to re-export SelectionProvider and StateCoordinator. Also add re-exports for PAFVProvider and DensityProvider (created by Plan 03 in the same wave — Plan 03 does not touch index.ts to avoid file ownership conflict).
