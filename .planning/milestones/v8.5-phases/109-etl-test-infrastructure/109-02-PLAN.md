# Test Environment Boundary Rules

## The Rule
Never import `realDb()` (or any sql.js WASM code) in a test file annotated with `@vitest-environment jsdom`. WASM and jsdom are incompatible runtimes.

## Rationale
- sql.js WASM requires a real Node.js environment to load the .wasm binary
- jsdom patches global `window`, `document`, `navigator` in ways that break WASM instantiation
- Mixing causes silent failures: WASM loads but produces corrupted results, or throws opaque "CompileError" at runtime
- The split is load-bearing -- it cannot be worked around with polyfills

## When to Use Each Environment

### Node (default, no annotation needed)
- Tests using `realDb()` from `tests/harness/realDb.ts`
- Tests using `makeProviders()` from `tests/harness/makeProviders.ts`
- Any test that touches sql.js Database, Worker handlers, or ETL pipeline
- Import/seam tests under `tests/seams/`

### jsdom (`// @vitest-environment jsdom` at top of file)
- Tests that need `document`, `window`, DOM APIs
- UI component tests (CommandBar, NotebookExplorer, ProjectionExplorer, etc.)
- Tests that create/query DOM elements
- Never import `realDb`, `makeProviders`, or `Database` in these files

## Correct Examples

```typescript
// CORRECT: Node environment (default) with realDb
import { realDb } from '../harness/realDb';
// No @vitest-environment annotation needed

describe('ETL import', () => {
  let db: Database;
  beforeEach(async () => { db = await realDb(); });
  afterEach(() => { db.close(); });
  // ...
});
```

```typescript
// CORRECT: jsdom environment for UI testing
// @vitest-environment jsdom
import { describe, it, expect } from 'vitest';

describe('CommandBar', () => {
  it('renders input', () => {
    const el = document.createElement('div');
    // ...
  });
});
```

## Incorrect Examples

```typescript
// WRONG: mixing realDb with jsdom -- WILL FAIL
// @vitest-environment jsdom
import { realDb } from '../harness/realDb'; // <-- CI will reject this
```

## CI Enforcement
A grep-based check in the CI pipeline scans all test files for violations. Any file containing both `realDb` (or `from.*Database`) and `@vitest-environment jsdom` will fail the build.

## Troubleshooting
- "CompileError: WebAssembly.instantiate()" -- check if jsdom annotation is present, remove it
- "document is not defined" -- you need `// @vitest-environment jsdom` but cannot use realDb in this file; split into two test files
- "Cannot find module 'sql-wasm.wasm'" -- SQL_WASM_PATH env not set; ensure tests/setup/wasm-init.ts globalSetup is configured

Add a new job `environment-boundary` (runs in parallel with existing jobs):

```yaml
  environment-boundary:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v5
      - name: Check WASM/jsdom boundary violations
        run: |
          VIOLATIONS=$(grep -rl '@vitest-environment jsdom' tests/ | xargs grep -l 'realDb\|from.*harness/realDb\|from.*database/Database' 2>/dev/null || true)
          if [ -n "$VIOLATIONS" ]; then
            echo "ERROR: WASM/jsdom boundary violation detected in:"
            echo "$VIOLATIONS"
            echo ""
            echo "These files mix @vitest-environment jsdom with realDb/Database imports."
            echo "See tests/ENVIRONMENT.md for the rule and how to fix."
            exit 1
          fi
          echo "No WASM/jsdom boundary violations found."
```

```typescript
mockPermission: (adapter: string, state: 'granted' | 'denied' | 'revoked') => {
  // Store permission state for TCC simulation
  const key = `__mock_permission_${adapter}`;
  if (state === 'revoked') {
    delete (window as any)[key];
  } else {
    (window as any)[key] = state;
  }
},
getPermissionState: (adapter: string): string | null => {
  const key = `__mock_permission_${adapter}`;
  return (window as any)[key] ?? null;
},
```

Also add cleanup in the `destroy()` method: iterate any `__mock_permission_*` keys and delete them.
