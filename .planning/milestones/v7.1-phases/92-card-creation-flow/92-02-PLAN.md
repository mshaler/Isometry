**Part C: Component-level Cmd+N handlers in NotebookExplorer.ts**

The ShortcutRegistry input guard means Cmd+N will not fire when the user is focused in a TEXTAREA or INPUT. Two component-level handlers are needed:

1. **Title input keydown handler** (`_titleKeydownHandler` in mount()): Add Cmd+N case. Currently handles Cmd+S and Escape. Add:
   ```typescript
   } else if ((e.metaKey || e.ctrlKey) && e.key === 'n') {
     e.preventDefault();
     if (this._creationState === 'buffering') {
       // Rapid creation: commit current name if non-empty, then fresh buffer
       const name = this._titleInputEl!.value.trim();
       if (name.length > 0) {
         void this._evaluateBufferingCommit().then(() => {
           // After commit and selection change, re-enter buffering for next card
           // Small delay to let selection change propagate
           this._enterBuffering();
         });
       } else {
         // Empty name: just restart buffering (no-op net effect)
         this._enterBuffering();
       }
     } else {
       // In editing state: auto-commit and enter buffering
       this.enterCreationMode();
     }
   }
   ```

2. **Textarea keydown handler** (`_keydownHandler` in mount(), the one attached to textarea): Add Cmd+N case. Currently handles Cmd+B/I/K/S. Add after the Cmd+S case:
   ```typescript
   } else if (e.key === 'n') {
     e.preventDefault();
     this.enterCreationMode();
   }
   ```
   Note: this handler already checks `const cmd = e.metaKey || e.ctrlKey;` and returns early if `!cmd`, so only meta-key combinations reach the switch. Adding `e.key === 'n'` is sufficient.
