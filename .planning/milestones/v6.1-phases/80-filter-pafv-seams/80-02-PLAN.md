**describe('CELL-02: __agg__ prefix regression guard'):**

6. Calc query with __agg__ prefix — call `handleSuperGridCalc(db, { rowAxes: [{ field: 'folder', direction: 'asc' }], where: 'deleted_at IS NULL', params: [], aggregates: { priority: 'sum', name: 'count' } })`. Verify:
   - Each result row has `groupKey.folder` (axis field, no prefix)
   - Each result row has `values.priority` (stripped from `__agg__priority`)
   - Each result row has `values.name` (stripped from `__agg__name`)
   - `values.priority` for Work folder = 60 (10+20+30)

7. Collision guard — use `priority` as BOTH a GROUP BY axis AND an aggregate target: `{ rowAxes: [{ field: 'priority', direction: 'asc' }], where: 'deleted_at IS NULL', params: [], aggregates: { priority: 'sum' } }`. Verify:
   - `groupKey.priority` is the raw group value (0, 5, 10, 15, 20, 30)
   - `values.priority` is the SUM (which equals the group value since each group has 1 card)
   - These do NOT collide (groupKey comes from axis column, values comes from `__agg__priority`)

8. Text field safety net — aggregate { name: 'sum' } (text field, should be coerced to 'count') → verify values.name is COUNT(*), not SUM(name)

**describe('CELL-03: hideEmpty flag'):**

NOTE: `hideEmpty` is NOT a parameter in SuperGridQueryConfig or handleSuperGridQuery. Read `src/worker/handlers/supergrid.handler.ts` and `src/views/supergrid/SuperGridQuery.ts` carefully. The `hideEmpty` behavior is implemented in the VIEW layer (SuperGrid.ts), not in the SQL query. The query always returns all non-empty groups (GROUP BY with COUNT(*) inherently excludes groups with 0 rows). The "empty" cells that hideEmpty controls are cross-product slots that have no matching data.

Test approach: Instead of testing the handler (which doesn't have hideEmpty), verify the SQL-level behavior — that GROUP BY only returns groups that actually have rows, not the full cross-product. This IS the correct seam for CELL-03: the SQL layer naturally excludes zero-count groups.

9. GROUP BY only returns existing combinations — With 2-axis config (card_type x folder), verify the returned cells do NOT include combinations that have no cards (e.g., event/Work, event/Personal do not appear because D1 has folder=null). Count cells returned vs theoretical cross-product (3 types x 3 folders = 9 possible, but only 5 returned).

10. Filter reduces groups — Add a WHERE filter for `folder = 'Work'`, verify only groups with Work cards appear (note/Work, task/Work).

**describe('CELL-04: sortOverrides produce ordered card_ids'):**

11. Default order (no sortOverrides) — 2-axis query for note/Work cell → card_ids contains A1 and A2 in some order (GROUP_CONCAT order is SQL-defined, typically insertion order).

12. sortOverrides by name DESC — `{ colAxes: [...], rowAxes: [...], where: 'deleted_at IS NULL', params: [], sortOverrides: [{ field: 'name', direction: 'desc' }] }` → for the note/Work cell, verify card_names are ['A2', 'A1'] (descending). NOTE: GROUP_CONCAT respects the query's ORDER BY, which includes sortOverrides after axis ordering.

13. sortOverrides by priority ASC — sortOverrides: [{ field: 'priority', direction: 'asc' }] → for note/Work cell, verify card order matches priority ascending (A1 at priority 10, A2 at priority 20).
