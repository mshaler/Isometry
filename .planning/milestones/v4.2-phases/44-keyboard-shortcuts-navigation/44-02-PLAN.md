```typescript
export interface ShortcutEntry {

  description: string; // e.g., 'List view', 'Undo'

export class ShortcutRegistry {
  register(shortcut: string, handler: () => void, meta: { category: string; description: string }): void;
  unregister(shortcut: string): void;
  getAll(): ShortcutEntry[];
  destroy(): void;
}
```


    2. Add 9 `.onReceive` handlers in the body, after the existing .onReceive handlers for undo/redo. Each one calls the existing `switchView(to:)` helper AND updates `selectedViewID` to keep the sidebar in sync:
    ```swift
    .onReceive(NotificationCenter.default.publisher(for: .switchToList)) { _ in
        selectedViewID = "list"
    }
    // ... repeat for all 9 views
    ```

    Note: We set `selectedViewID` which triggers `.onChange(of: selectedViewID)` which calls `switchView(to:)`. This keeps the sidebar selection and the web view in sync. Do NOT call switchView directly from onReceive — let onChange handle it to avoid the duplication.

    **In IsometryApp.swift:**

    1. Add a `ViewCommands` struct alongside the existing `IsometryCommands`. Use `CommandMenu("View")` to create a new top-level View menu:

    ```swift
    struct ViewCommands: Commands {
        var body: some Commands {
            CommandMenu("View") {
                Button("List") {
                    NotificationCenter.default.post(name: .switchToList, object: nil)
                }
                .keyboardShortcut("1", modifiers: .command)

                Button("Grid") {
                    NotificationCenter.default.post(name: .switchToGrid, object: nil)
                }
                .keyboardShortcut("2", modifiers: .command)

                // ... continue for all 9 views:
                // 3=Kanban, 4=Calendar, 5=Timeline, 6=Gallery, 7=Network, 8=Tree, 9=SuperGrid
            }
        }
    }
    ```

    2. Register ViewCommands alongside IsometryCommands in the WindowGroup:
    ```swift
    .commands {
        IsometryCommands()
        ViewCommands()
    }
    ```

    IMPORTANT: The view order for Cmd+1-9 MUST match the JS side:
    1=List, 2=Grid, 3=Kanban, 4=Calendar, 5=Timeline, 6=Gallery, 7=Network, 8=Tree, 9=SuperGrid

    Note: In the native app, both the Swift menu bar Cmd+1-9 AND the JS ShortcutRegistry Cmd+1-9 will fire. This is fine — the SwiftUI menu sends a notification that sets selectedViewID which triggers switchView, and the JS registry calls viewManager.switchTo directly. Both paths converge to the same result. The SwiftUI menu takes priority (captures the shortcut before WKWebView sees it), so the JS handler won't fire in the native app. This is the correct behavior — native menu owns the shortcut in native, JS owns it in web dev mode.
