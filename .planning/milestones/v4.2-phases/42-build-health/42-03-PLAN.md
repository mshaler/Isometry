<task type="auto">
  <name>Task 1: Create GitHub Actions CI workflow</name>
  <files>.github/workflows/ci.yml</files>
  <action>
1. Create the `.github/workflows/` directory.

2. Create `.github/workflows/ci.yml` with three parallel jobs per user decisions:

   **Trigger:** Every push to any branch (`on: push: branches: ['**']`)

   **Job 1 - typecheck:**
   - runs-on: ubuntu-latest
   - steps: checkout@v5, setup-node@v6 (node-version: 22, cache: npm), npm ci, npx tsc --noEmit

   **Job 2 - lint:**
   - runs-on: ubuntu-latest
   - steps: checkout@v5, biomejs/setup-biome@v2 (version: latest), biome ci .
   - Note: This job does NOT need npm ci because biomejs/setup-biome installs the biome binary directly. Use `biome ci .` (not `biome check .`) -- the `ci` command is designed for CI and provides cleaner output with non-zero exit on any issue.

   **Job 3 - test:**
   - runs-on: ubuntu-latest
   - steps: checkout@v5, setup-node@v6 (node-version: 22, cache: npm), npm ci, npx vitest --run

   All three jobs are independent (no `needs` between them) so they run in parallel. If any job fails, the push is marked as failed (fail-fast per user decision).

   Pin Node to version 22 (LTS) per RESEARCH.md Pitfall 4 recommendation.
   Use actions/checkout@v5 and actions/setup-node@v6 per RESEARCH.md State of the Art.

3. Verify the workflow file is valid YAML and follows GitHub Actions schema. The `name` should be "CI".
  </action>
  <verify>
    <automated>python3 -c "import yaml; yaml.safe_load(open('.github/workflows/ci.yml'))" && echo "Valid YAML"</automated>
  </verify>
  <done>.github/workflows/ci.yml exists with 3 parallel jobs (typecheck, lint, test), triggers on push to all branches, uses correct action versions and Node 22.</done>
</task>

<task type="auto">
  <name>Task 2: Push workflow and configure branch protection</name>
  <files>.github/workflows/ci.yml</files>
  <action>
1. After the CI workflow file is committed and pushed, wait for the first CI run to complete. Check status with:
   ```
   gh run list --limit 5
   ```

2. Once all three jobs pass, configure branch protection on `main` using the gh CLI:
   ```
   gh api repos/{owner}/{repo}/branches/main/protection \
     --method PUT \
     --input - << 'EOF'
   {
     "required_status_checks": {
       "strict": true,
       "contexts": ["typecheck", "lint", "test"]
     },
     "enforce_admins": false,
     "required_pull_request_reviews": null,
     "restrictions": null
   }
   EOF
   ```

   Per RESEARCH.md Pitfall 5: branch protection can only reference status checks that have run at least once. That's why this step must wait for the first CI run.

   If branch protection fails (e.g., repo permissions), note it in the SUMMARY as a manual step for the user to configure in GitHub Settings > Branches > main > Require status checks.

3. Verify branch protection is set:
   ```
   gh api repos/{owner}/{repo}/branches/main/protection
   ```
  </action>
  <verify>
    <automated>gh run list --limit 1 --json conclusion --jq '.[0].conclusion'</automated>
  </verify>
  <done>CI workflow runs successfully on push. Branch protection on main requires typecheck, lint, and test to pass. BUILD-05 complete.</done>
</task>
