   - The virtualizer counts LEAF ROWS (unique row axis values), not total cards or cells
   - For a dual-axis PAFV projection (folder x card_type) at 20K cards:
     - folders: ~5 unique values, card_types: 5 unique values
     - If folder is row axis: 5 leaf rows (well below VIRTUALIZATION_THRESHOLD=100)
     - The virtualizer would NOT activate — the bottleneck is cell count per row, not row count
   - For a single-axis projection (created_at month) at 20K cards:
     - ~12 months of data → 12 rows (below 100)
   - For a high-cardinality axis (folder with many values): could exceed 100

2. **Decision point (Claude's Discretion per CONTEXT.md):**
   - If profiling shows row count stays below 100 for typical PAFV projections at 20K:
     - Keep VIRTUALIZATION_THRESHOLD=100 (correct — virtualizer isn't the fix for dual-axis render)
     - Document: "Virtualizer activates for high-cardinality single-axis projections; dual-axis render bottleneck is header span computation, not row count"
   - If any test shows row count > 100: virtualizer already activates correctly

3. **OVERSCAN_ROWS tuning:**
   - The current value of 5 is reasonable for 28px row heights
   - At 20K with high-cardinality axis: 5 overscan rows = 5 * 28px = 140px buffer

   - Per CONTEXT.md: "test values 3, 5, 10" — since jsdom can't measure flicker, keep 5 and document the decision
