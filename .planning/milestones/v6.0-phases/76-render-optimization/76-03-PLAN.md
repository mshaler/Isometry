  <name>Task 1: Profile and optimize _renderCells for multi-axis rendering</name>
  <files>src/views/SuperGrid.ts</files>
  <action>
1. **Profile where _renderCells time is spent at dual-axis 5K cells:**
   Add temporary performance.now() markers inside _renderCells to identify the dominant cost:
   - Header computation (colHeaders/rowHeaders derivation from cells)
   - D3 data join (selectAll + data + enter + exit)
   - DOM element creation (div creation, class assignment, text content)
   - CSS Grid style assignment (gridRow/gridColumn computation)

   Run budget-render.test.ts with console.log timing to identify which phase dominates.

2. **Apply targeted optimizations based on profiling results:**

   Likely bottlenecks and fixes (apply whichever profiling identifies):

   a. **Header span computation (most likely bottleneck for multi-axis):**
      - The colHeaders/rowHeaders derivation iterates all cells to compute run-length spans
      - For dual-axis 5K cells: this is O(cells * axes) repeated for both col and row
      - Fix: Memoize or pre-compute header structure. If cells haven't changed (same axis config), reuse the previous header computation
      - Cache key: hash of axis fields + sort of unique axis values

   b. **D3 data join with large cell count:**
      - At 5K cells, D3 selectAll().data().join() creates 5K DOM elements
      - Fix: If virtualizer is not active (row count < threshold), apply a cell-count-based windowing:
        - For cells > 1000: only render cells in the viewport (estimate from container size)
        - Use a lightweight cell virtualizer that filters cells before the D3 join

   c. **Redundant DOM reads in the render loop:**
      - gridRow/gridColumn computed per-cell may trigger layout thrashing
      - Fix: Batch all style writes using DocumentFragment or by building HTML string

   d. **Header DOM reconstruction on every render:**
      - If headers are rebuilt from scratch each time, this is O(uniqueValues * axes)
      - Fix: D3 data-join the headers (enter/update/exit) instead of removing+recreating

3. **Key constraint: Do NOT break existing functionality.**
   - Keep D3 key functions on all .data() calls
   - Do NOT register event listeners inside _renderCells
   - Do NOT modify the virtualizer (Plan 02 handles that)
   - Preserve all existing CSS class assignments and data attributes

4. **If 128ms jsdom budget is unachievable for dual-axis 5K:**
   Per CONTEXT.md: "keep original targets unless truly impossible"
   - Try all optimizations first
   - If still > 128ms after all optimizations, investigate whether the test's synthetic data (makeSyntheticCells) creates a worst-case that real data wouldn't hit
   - Only as last resort: adjust BUDGET_RENDER_JSDOM_MS with documented rationale in PerfBudget.ts
   - ANY budget adjustment must be accompanied by measurement data showing the new budget is the practical floor

5. **Run budget-render.test.ts after each optimization to measure impact.**
