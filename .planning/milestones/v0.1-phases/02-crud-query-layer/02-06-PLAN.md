From tsconfig.json (root cause of Gap 3):
```json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true
  }
}
```

The `noUncheckedIndexedAccess` option causes array element access (`arr[i]`) to return `T | undefined` instead of `T`. All 9 TypeScript errors in search.test.ts stem from this setting — array element accesses need non-null assertions (`!`) or explicit null guards after a length check that already guarantees the element exists.

<task type="auto">
  <name>Task 3: Fix TypeScript strict-mode errors in search.test.ts</name>
  <files>tests/database/search.test.ts</files>
  <action>
Fix 9 TypeScript errors (TS2532/TS18048) caused by `noUncheckedIndexedAccess: true` in tsconfig.json. All errors are in `tests/database/search.test.ts` where array elements are accessed without null guards despite prior `expect(results.length).toBeGreaterThan(N)` checks.

**Fix 1 — Line 78 (rank sort comparison):**
The loop body `results[i - 1].rank` and `results[i].rank` — both potentially undefined.

Change:
```typescript
      expect(results[i - 1].rank).toBeLessThanOrEqual(results[i].rank);
```
To:
```typescript
      expect(results[i - 1]!.rank).toBeLessThanOrEqual(results[i]!.rank);
```

Justification: The loop runs `i = 1..length-1`, and `results.length > 1` is asserted on line 76. Both `results[i-1]` and `results[i]` are guaranteed defined.

**Fix 2 — Lines 106-112 (result field checks):**
`const result = results[0]` is `SearchResult | undefined`. Lines 107-112 access `result.card`, `result.card.id` etc.

Change:
```typescript
    const result = results[0];
    expect(result.card).toBeDefined();
    expect(typeof result.card.id).toBe('string');
    expect(result.card.id).toMatch(/^[0-9a-f-]{36}$/);
    expect(typeof result.card.name).toBe('string');
    expect(typeof result.card.created_at).toBe('string');
    expect(Array.isArray(result.card.tags)).toBe(true);
```
To:
```typescript
    const result = results[0]!;
    expect(result.card).toBeDefined();
    expect(typeof result.card.id).toBe('string');
    expect(result.card.id).toMatch(/^[0-9a-f-]{36}$/);
    expect(typeof result.card.name).toBe('string');
    expect(typeof result.card.created_at).toBe('string');
    expect(Array.isArray(result.card.tags)).toBe(true);
```

Justification: Line 105 asserts `results.length > 0`, so `results[0]` is guaranteed defined.

**Fix 3 — Line 167 (snippet truthy check):**
`results[0].snippet` — `results[0]` is potentially undefined.

Change:
```typescript
    expect(results[0].snippet).toBeTruthy();
```
To:
```typescript
    expect(results[0]!.snippet).toBeTruthy();
```

Justification: Line 165 asserts `results.length > 0`, so `results[0]` is guaranteed defined.

**Summary:** All fixes add non-null assertions (`!`) to array element accesses that are already guarded by prior `expect(length > N)` assertions. This is correct because:
- The `expect` on length will throw and fail the test if the array is empty
- The `!` tells TypeScript "I know this is defined because of the runtime check above"
- No behavioral change — all 21 search tests continue to pass identically

Do NOT change any test logic, assertions, or add/remove tests.
