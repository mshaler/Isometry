<objective>
Sync the native SwiftUI shell appearance with the web content theme, adding a theme picker to SettingsView and injecting the saved theme before first paint to prevent FOWT.

Purpose: Without native sync, the SwiftUI sidebar/toolbar would stay in system default appearance while the WKWebView content switches themes — a jarring visual disconnect. The WKUserScript injection prevents the dark flash that would otherwise appear when the user has selected light mode.

Output: SettingsView with Appearance picker, ContentView with .preferredColorScheme() and theme injection, bidirectional theme sync between Swift and JS.
</objective>

  <files>native/Isometry/Isometry/SettingsView.swift, native/Isometry/Isometry/ContentView.swift</files>
  <action>
1. **SettingsView.swift** — Add Appearance section with 3-way theme picker:

   Add `@AppStorage("theme") private var theme: String = "dark"` property.

   Add a new Section between "Subscription" and "About":
   ```swift
   // MARK: Appearance
   Section {
       Picker("Appearance", selection: $theme) {
           Text("Light").tag("light")
           Text("Dark").tag("dark")
           Text("System").tag("system")
       }
       .pickerStyle(.segmented)
   } header: {
       Text("Appearance")
   }
   ```

   The `.segmented` style gives a clean 3-way toggle. `@AppStorage("theme")` automatically persists to UserDefaults.

2. **ContentView.swift** — Add `.preferredColorScheme()` modifier and theme sync:

   a. Add `@AppStorage("theme") private var theme: String = "dark"` property.

   b. Add computed property for ColorScheme:
   ```swift
   private var preferredScheme: ColorScheme? {
       switch theme {
       case "light": return .light
       case "dark": return .dark
       default: return nil  // "system" — use system default
       }
   }
   ```

   c. Apply `.preferredColorScheme(preferredScheme)` modifier on the NavigationSplitView. This controls the SwiftUI sidebar, toolbar, and status bar appearance.

   d. Add `.onChange(of: theme)` handler to push theme to JS:
   ```swift
   .onChange(of: theme) { _, newTheme in
       let js = "window.__isometry?.themeProvider?.setTheme('\(newTheme)')"
       Task { try? await bridgeManager.webView?.evaluateJavaScript(js) }
   }
   ```

   e. In `setupWebView()`, inject a WKUserScript at `.atDocumentStart` to set data-theme BEFORE CSS loads:
   ```swift
   let savedTheme = UserDefaults.standard.string(forKey: "theme") ?? "dark"
   let themeScript = WKUserScript(
       source: "document.documentElement.setAttribute('data-theme', '\(savedTheme)');document.documentElement.className='no-theme-transition';",
       injectionTime: .atDocumentStart,
       forMainFrameOnly: true
   )
   config.userContentController.addUserScript(themeScript)
   ```
   Add this BEFORE the consoleScript registration so it runs as early as possible.

IMPORTANT: The `@AppStorage("theme")` key must match the key used in `UserDefaults.standard.string(forKey: "theme")` in the WKUserScript injection. Both use "theme" as the key.

IMPORTANT: Do NOT read from `@AppStorage` inside setupWebView() — use `UserDefaults.standard.string(forKey:)` directly because setupWebView() runs before SwiftUI property wrappers are fully initialized in all lifecycle scenarios.
