## States and Interactions

### ListView (VIEW-01)

**Primary visual anchor:** The card list column — the full-width stack of `.card` rows is the focal point. The sort toolbar above it is secondary UI chrome.

**Normal render state:**
- Single-column list of `.card` HTML elements via `renderDimensionCard()`
- Sort toolbar above list: `<select>` for field (name / created_at / modified_at / priority), toggle button for asc/desc direction
- Sort direction toggle button: glyph "↑" or "↓" with `aria-label="Sort ascending"` / `aria-label="Sort descending"` and `title` attribute matching
- [data-dimension] attribute on container controls 1x/2x/5x visual density
- D3 key function `d => d.id` mandatory on every `.data()` call
- Double-click or Enter on focused card opens 10x detail overlay

**Empty state (VIEW-01 fix, no data):**
- Container: `.view-empty` → `.view-empty-panel`
- Icon: Unicode list glyph `☰` at 32px via `.view-empty-icon`
- Heading: "No cards to list" — `--text-lg` (16px), weight 600
- Description: "Import data to see your cards in list view." — `--text-base` (13px), weight 400
- No CTA button (welcome state handled by ViewManager; filtered-empty shows Clear Filters)

**Filtered-empty state:**
- Same `.view-empty-panel` with a `.clear-filters-btn` "Clear Filters" button below description

**Error state:**
- `.view-error-banner` — "List view failed to load — database query error. Try reloading." + `.retry-btn`

**Keyboard navigation (A11Y-08):**
- Arrow Up/Down moves focus through card list
- Enter or double-click opens card detail
- Composite widget: container has `tabindex="0"`, individual cards get `tabindex="-1"` + `.card--focused` class on focus

### GridView (VIEW-02)

**Primary visual anchor:** The CSS Grid of card tiles — the mosaic of `.card` elements filling the container is the focal point.

**Normal render state:**
- CSS Grid of `.card` HTML tiles via `renderDimensionCard()`
- Column count = `Math.max(1, Math.floor(containerWidth / 180))`
- [data-dimension] attribute on container controls visual density
- D3 key function `d => d.id` mandatory

**Empty state (VIEW-02 fix, no data):**
- Icon: Unicode grid glyph `▦` (U+25A6) at 32px via `.view-empty-icon`
- Heading: "No cards to display"
- Description: "Import data to see cards arranged in a grid."

**Keyboard navigation (A11Y-08):**
- Arrow keys navigate grid 2D (Left/Right within row, Up/Down across rows)
- Enter or double-click opens card detail

### KanbanView (VIEW-03)

**Primary visual anchor:** The column headers row — the horizontal stripe of named `.kanban-column-header` elements establishes the grouping structure and is the first visual element users scan.

**Normal render state:**
- Horizontal flex of `.kanban-column` elements, each with header + scrollable body
- Cards in each column rendered via `renderDimensionCard()` with pointer-event drag-drop
- Column domain derived from `groupByField` values in the query result (default: `status`)
- Empty columns for domain values with zero cards still render (show column header with 0 count)
- Ghost card (`.kanban-card--ghost`) follows cursor during drag; source card dims to `opacity: 0.4`
- Drop target column body gets `.drag-over` class during drag-over

**Empty state (VIEW-03 fix, no data):**
- Icon: `⁞⁞⁞` (three vertical bars, U+2759 × 3) at 32px via `.view-empty-icon`
- Heading: "No cards to organize"
- Description: "Import data to see cards sorted by status columns."

**Drag-drop visual contract:**
- Ghost card: `position: fixed`, `pointer-events: none`, `z-index: 9999`, `opacity: 0.8`, `transform: scale(1.05)`
- Drag-over column: `outline: 1px solid var(--accent)`, `outline-offset: -2px`
- Source card during drag: `opacity: 0.4`
- Column highlight must clear on `pointerup` (no lingering `.drag-over` after drop)

### CalendarView (VIEW-04)

**Primary visual anchor:** The month grid of day cells — the calendar grid is the focal point. Navigation controls (prev/next, period label) are secondary chrome above the grid.

**Normal render state:**
- CSS Grid calendar layout with weekday header row + day cells
- Cards with `due_at` rendered as chips inside their matching day cell (max 2 chips + "+N more" overflow)
- Cards with `due_at = null` are excluded — this is expected behavior, not an error
- Navigation: prev/next buttons for month/week/quarter/year; period label centered between
- Granularity toggle driven by DensityProvider (month/week/day/quarter/year)
- Today's date cell receives `var(--accent-bg)` background tint

**Empty state (VIEW-04 fix, no data):**
- If zero cards are returned from ViewManager:
  - Icon: `📅` (U+1F4C5) at 32px via `.view-empty-icon`
  - Heading: "No dated cards"
  - Description: "Cards need due dates to appear on the calendar."
- Note: cards exist but all have `due_at = null` renders calendar structure with empty cells — NOT the empty panel. The empty panel is for zero-card DB result only.

**Filtered-empty state:**
- Shows `.clear-filters-btn` below description when filters are active

### GalleryView (VIEW-05)

**Primary visual anchor:** The tile grid — the mosaic of `.gallery-tile` elements, each with a large type icon or image, is the focal point.

**Normal render state:**
- CSS Grid of `div.gallery-tile` elements sized 240×160px
- Resource cards (`card_type = 'resource'`): `img.tile-image` with `body_text` as src; `onerror` shows fallback icon
- All other cards: large type icon character (`CARD_TYPE_ICONS`) centered + card name below
- Column count = `Math.max(1, Math.floor(containerWidth / 240))`
- D3 key function `d => d.id` mandatory

**Empty state (VIEW-05 fix, no data):**
- Icon: `🖼` (U+1F5BC) at 32px via `.view-empty-icon`
- Heading: "No cards to browse"
- Description: "Import data to see cards in gallery view."

**Keyboard navigation (A11Y-08):**
- Arrow keys navigate grid 2D
- Enter or double-click opens card detail
- `.gallery-tile--focused` class applied to focused tile

### TreeView (VIEW-06)

**Primary visual anchor:** The D3 hierarchy SVG — the tree of connected nodes filling the SVG viewport is the focal point. The orphan section below it is secondary.

**Normal render state:**
- SVG-based collapsible hierarchy via `d3-hierarchy` + `d3.tree()` top-down vertical layout
- Nodes as `<circle>` with radius 16px, colored by `card_type` (see Color section)
- Node labels as SVG `<text>` at 11px below each circle
- Edges as SVG `<path>` curved links (`d3.linkVertical()`) with `var(--border-muted)` stroke
- Collapse/expand toggles `_children` stash WITHOUT re-stratifying (locked decision)
- Orphan cards (no hierarchy connections) rendered in a separate HTML div below SVG
- Zoom/pan via `d3.zoom()` on SVG root
- D3 key function `d => d.data.id` mandatory

**Empty state (VIEW-06 fix, no data):**
- If zero cards returned from ViewManager:
  - Icon: `🌳` (U+1F333) at 32px via `.view-empty-icon`
  - Heading: "No hierarchy found"
  - Description: "Import data with folder structure to see the tree."
- Note: cards with no hierarchical connections render as orphan cards — NOT an empty state

**Orphan section (valid state — not empty):**
- Separated by `var(--border-subtle)` horizontal rule below SVG
- Section label: "Ungrouped cards" at `--text-sm` (11px), weight 600, `var(--text-secondary)`
- Orphan cards rendered as HTML `.card` elements in single-column layout

### Cross-View UX (CVUX-01, CVUX-02)

#### View Switching (CVUX-01)

Three entry points must all produce identical `switchTo()` behavior:

1. **SidebarNav click** — clicking a `.sidebar-item` in the Visualization Explorer section
   - Immediate visual feedback: `.sidebar-item--active` class applied synchronously before async render
   - Active state: background `var(--accent-bg)`, left border `var(--accent)` 2px solid, text `var(--accent)`, label font-weight 600
   - Previous active item loses `.sidebar-item--active` synchronously

2. **Keyboard shortcut (Cmd+1–9)** — `ShortcutRegistry` maps `Cmd+1` through `Cmd+9` to view types
   - View order: 1=list, 2=grid, 3=kanban, 4=calendar, 5=gallery, 6=tree, 7=network, 8=supergrid, 9=timeline
   - SidebarNav `setActiveItem()` must be called after view switch to keep sidebar in sync
   - No visual state change on the keyboard shortcut itself — only the destination view changes

3. **Command palette** — selecting a "Switch to X view" entry
   - Same `switchTo()` call path; SidebarNav `setActiveItem()` synced after palette closes

**View transition contract:**
- Current view `destroy()` called before next view `mount()`
- `crossfadeTransition` or `shouldUseMorph` (from `src/views/transitions.ts`) applied between views
- `.view-loading` shown after 200ms delay if render takes longer
- `pafv.setViewType()` called inside `switchTo()` to apply VIEW_DEFAULTS

#### Empty States Per View (CVUX-02)

All 6 views must respect the three-tier empty state hierarchy already established in `ViewManager`:

| Tier | Trigger | What shows |
|------|---------|-----------|
| Welcome | Zero cards in database | Welcome panel with "Load Sample Data" CTA + import buttons |
| Filtered-empty | Cards exist but query returns 0 rows due to active filters | View-specific panel + `.clear-filters-btn` "Clear Filters" |
| View-specific | Cards exist, no filters, but view has a structural requirement (e.g. CalendarView needs `due_at`) | View-specific panel with contextual explanation |

**Welcome panel:** Shared across all 9 views — unchanged from Phase 43. No modifications needed.

**Filtered-empty panel per view:** Existing `VIEW_EMPTY_MESSAGES` in `ViewManager.ts` lines 48–80 already defines all 9 view messages. Phase must ensure the empty state actually renders by verifying `ViewManager._fetchAndRender()` is correctly wired to all 6 views.

## Data Render Contracts

### Shared ViewManager path (all 6 views)

All 6 views route through `ViewManager._fetchAndRender()`. The data path contract:

```
StateCoordinator subscription fires
  → ViewManager._fetchAndRender()
      → QueryBuilder builds SQL from active providers
      → WorkerBridge db:exec → rows
      → toCardDatum(row) for each row
      → view.render(cards)
      → If cards.length === 0: show empty panel
      → Else: show view output
```

This differs from SuperGrid (BridgeDataAdapter) and Timeline/Network (secondary queries). Phase must diagnose and fix any break in this pipeline for each of the 6 views.

### KanbanView special case

KanbanView requires `MutationManager` for drag-drop. It must be injected at construction time by whoever creates the view instance (verified in `src/views/KanbanView.ts`). The `groupByField` defaults to `'status'`.

### TreeView special case

TreeView needs a secondary `db:exec` for connection rows to build the hierarchy. The data path:

```
1. Primary db:exec → card rows → stratify into hierarchy
2. Secondary db:exec → connection rows (type = groupByField) → build parent-child relationships
3. d3.tree() layout → SVG render
4. Orphan cards (no connections) → HTML list render
```

If the connections query returns zero rows, all cards render as orphans — valid state, no empty panel.

### CalendarView special case

CalendarView constructs its own time-period grid structure independently of the card data. If zero cards come from ViewManager, the calendar grid still renders (empty cells) — only show `.view-empty-panel` if the ViewManager query itself returns zero rows (welcome or filtered-empty state). The view-specific "No dated cards" message shows when all returned cards have `due_at = null` — detected internally by CalendarView, not by ViewManager.
